CPython Runtime · 01
CPython VM & Bytecode
Your Python source goes through 4 stages before a single instruction executes: tokenisation → parsing → compilation → evaluation. Understanding this pipeline explains every "magic" behaviour in Python.
🔬Stage 1–3: Source → AST → BytecodeINTERNALS

Tokenisation (tokenize module): The source string is converted into a flat stream of tokens — NAME, NUMBER, STRING, INDENT, DEDENT, NEWLINE, OP. INDENT/DEDENT tokens are generated by tracking indentation levels. This is why mixing tabs/spaces is a tokeniser-level error.

Parsing: Tokens are fed into the parser (CPython 3.9+ uses a PEG parser, before that a LL(1) grammar). The parser builds a Concrete Syntax Tree (CST), then immediately converts it to an Abstract Syntax Tree (AST). Each AST node is a Python object — you can inspect it with import ast; ast.dump(ast.parse("x=1+2")).

Compilation: The compile() built-in converts the AST to a code object (types.CodeType). A code object contains: co_code (raw bytecode bytes), co_consts (constants pool), co_names (global/attr names), co_varnames (local variable names), co_filename, co_firstlineno, and more. This is what gets stored in .pyc files.

SOURCE CODE PIPELINE ─────────────────────────────────────────────────────────────── "x = 1 + 2" │ ▼ tokenize module [NAME 'x'] [OP '='] [NUMBER '1'] [OP '+'] [NUMBER '2'] │ ▼ PEG parser (Python 3.9+) AST: Module(body=[Assign( targets=[Name(id='x')], value=BinOp(left=Constant(1), op=Add(), right=Constant(2)))]) │ ▼ compile() → code object CodeObject: co_code: b'd\x00d\x01\x17\x00Z\x00d\x02S\x00' co_consts: (1, 2, None) co_varnames: () co_names: ('x',) │ ▼ CPython eval loop (ceval.c) LOAD_CONST 1 push 1 onto stack LOAD_CONST 2 push 2 onto stack BINARY_ADD pop 2, pop 1, push 3 STORE_NAME 'x' pop 3, store in globals['x']
⚙️Stage 4: The CPython Evaluation Loop (ceval.c)INTERNALS

The heart of CPython is a giant C switch statement in Python/ceval.c. The main loop dispatches on each opcode, which is just a 1-byte integer. CPython uses a stack-based virtual machine — there are no registers. All operations push/pop from a value stack stored in the frame.

A frame object (PyFrameObject) is created for every function call. It stores: the code object, a reference to the global namespace dict, the local variables array, the value stack, and the current instruction pointer. Frames are chained to form the call stack — sys._current_frames() returns them.

Python 3.11 introduced specialised adaptive bytecode: after an instruction is executed a few times, the interpreter inspects what types it saw and replaces the generic opcode with a faster specialised version (e.g. LOAD_ATTR → LOAD_ATTR_INSTANCE_VALUE). This is CPython's equivalent of JIT warm-up.

inspect_bytecode.py

💡 dis module: dis.dis(fn) is your window into what Python actually executes. When you wonder "why is this slow?", disassemble it. You may find surprising things — attribute lookups, function calls, and global variable access all have different opcodes with different costs. LOAD_FAST (local) is faster than LOAD_GLOBAL, which is faster than LOAD_ATTR.
CPython Runtime · 02
The GIL — Global Interpreter Lock
The GIL is the most misunderstood feature in Python. It is not a language feature — it's a CPython implementation detail. Here's exactly what it does and doesn't protect.
🔒What the GIL Actually IsINTERNALS

The GIL is a mutex (mutual exclusion lock) that CPython holds whenever it executes Python bytecode. Only one thread can hold the GIL at a time, meaning only one thread can execute Python bytecode at any given moment — even on a multi-core CPU.

Why does it exist? CPython's memory management is not thread-safe. The reference counting system (every object has a ob_refcnt field) requires atomic increments and decrements. Without the GIL, two threads could simultaneously modify the same refcount, leading to use-after-free bugs or memory leaks. The GIL is a coarse-grained solution that makes the entire interpreter safe without per-object locks.

What it protects: Reference counts, the garbage collector, the import system, and internal Python data structures. What it doesn't protect: Your own data — if two threads modify the same Python list, you can still get data races at the Python level (though not at the C level). The GIL is released and reacquired frequently.

GIL: CPU TIME ALLOCATION WITH 2 THREADS ────────────────────────────────────────────────────────── Thread 1 (CPU-bound): ████████░░░░░░░░████████░░░░░░░░████████ Thread 2 (CPU-bound): ░░░░░░░░████████░░░░░░░░████████░░░░░░░░ │◄─ sys.getswitchinterval() = 5ms ─►│ └── GIL check interval, then forced switch Effective CPU usage: ~1 core (serial execution, not parallel) GIL: CPU TIME WITH I/O-BOUND THREADS ────────────────────────────────────────────────────────── Thread 1 (network): ████░░░░░░░░░░░░████░░░░░░░░░░░░████ Thread 2 (network): ░░░░████░░░░░░░░░░░░████░░░░░░░░░░░░ │ GIL released during I/O wait ──── threads CAN overlap here Effective: genuine concurrency for I/O bound work ✓ C EXTENSIONS (NumPy, etc.): ────────────────────────────────────────────────────────── Thread 1 (np.dot): ██ ─── releases GIL ───→ ████████████████ Thread 2 (Python): ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ NumPy releases the GIL during C computation → true parallelism
🧵GIL Release Points & What Actually Runs in ParallelMENTAL MODEL

The GIL is released in several situations:

  • I/O operations: Any blocking I/O call (socket.recv(), file.read(), time.sleep()) releases the GIL. The thread blocks in the kernel, allowing other threads to run Python code.
  • C extensions: Any C extension can call Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS to release the GIL around its computation. NumPy, scipy, OpenBLAS, and most numeric libraries do this. This is why np.dot() on a large matrix runs in parallel across threads.
  • Switch interval: Every sys.getswitchinterval() seconds (default 5ms), the current thread is forced to release the GIL, giving other threads a chance to acquire it.

Python 3.13 adds a no-GIL build (PEP 703, experimental). Instead of a global lock, each object has its own per-object locking and biased reference counting. This is the biggest CPython change in years.

gil_demonstration.py
import threading, time import numpy as np # CPU-bound threads — GIL prevents true parallelism def count(n): total = 0 for _ in range(n): total += 1 return total t0 = time.time() count(10_000_000) count(10_000_000) print(f"serial: {time.time()-t0:.2f}s") t0 = time.time() t1 = threading.Thread(target=count, args=(10_000_000,)) t2 = threading.Thread(target=count, args=(10_000_000,)) t1.start(); t2.start(); t1.join(); t2.join() print(f"threads: {time.time()-t0:.2f}s") # ≈ same or SLOWER — GIL! # NumPy releases the GIL — threads DO run in parallel big = np.random.randn(5000, 5000) t0 = time.time() def numpy_work(): np.dot(big, big) # GIL released inside C code t1 = threading.Thread(target=numpy_work) t2 = threading.Thread(target=numpy_work) t1.start(); t2.start(); t1.join(); t2.join() print(f"numpy threads: {time.time()-t0:.2f}s") # faster than 2x serial! # Fix for CPU-bound: use multiprocessing (separate processes, separate GILs) from multiprocessing import Pool with Pool(4) as p: results = p.map(count, [5_000_000] * 4) # 4x speedup on 4 cores
CPython Runtime · 03
Memory Management & Garbage Collection
CPython uses reference counting as the primary memory management strategy, with a cyclic garbage collector as a fallback. Understanding this explains del, weakref, and when objects are freed.
♻️Reference Counting: The Primary MechanismINTERNALS

Every Python object starts with a C struct header containing two fields: ob_refcnt (reference count) and ob_type (pointer to the type object). When you bind a name to an object, its refcount increments. When the binding goes away (variable reassigned, out of scope, deleted), it decrements. When refcount hits zero, tp_dealloc is called immediately — the object is freed right then, deterministically.

This is why Python objects have deterministic destruction: a file is closed the instant all references drop. Unlike Java's GC, you know exactly when __del__ fires (for non-cyclic objects). The with statement leverages this — when the block exits, the __exit__ is called because the context manager's refcount drops.

The problem: reference cycles. If object A holds a reference to B, and B holds a reference to A, both have refcount ≥ 1 even if nothing else references them. Pure reference counting can never free these — they'd leak forever.

REFERENCE COUNTING IN ACTION ────────────────────────────────────────────────────────────── a = MyObject() # ob_refcnt = 1 b = a # ob_refcnt = 2 (a and b both reference it) del a # ob_refcnt = 1 (a removed from namespace) b = None # ob_refcnt = 0 ← tp_dealloc() called NOW REFERENCE CYCLE (refcount can never reach 0) ────────────────────────────────────────────────────────────── a = {}; b = {} # a.refcnt=1, b.refcnt=1 a['b'] = b # b.refcnt=2 b['a'] = a # a.refcnt=2 del a, b # a.refcnt=1, b.refcnt=1 ← STUCK! # Nothing else references them # Pure refcount can't free them → needs cyclic GC CYCLIC GC (gc module) ────────────────────────────────────────────────────────────── Objects are tracked in 3 "generations" (gen 0, 1, 2) Generation 0: new objects — collected most frequently Generation 2: long-lived objects — collected rarely Mark phase: traverse references, compute "effective refcount" Sweep phase: anything with effective refcount 0 is unreachable → free
🧊pymalloc: CPython's Custom AllocatorDEEP DIVE

CPython doesn't call malloc() for every small object. It has pymalloc, a custom arena-based allocator for objects ≤ 512 bytes. pymalloc organises memory as: Arenas (256 KB each, aligned to 256KB) → Pools (4KB pages) → Blocks (fixed-size units per pool). Each pool serves one specific block size class (8, 16, 24, ..., 512 bytes).

This structure means allocating a small Python object is nearly free — just grabbing a block from a free-list, no system call, no heap fragmentation. It's a major performance win since Python creates millions of tiny objects.

Object caching: CPython caches small integers (-5 to 256) and interned strings at the interpreter level. id(42) == id(42) is always true because there's only one int object for 42. This is why is comparison works for small integers in practice — but you should never rely on it.

CPython Runtime · 04
The Python Object Model
In CPython, every Python object is a C struct. The type system is itself implemented in Python's object model — type is an instance of type, and object is the base of everything.
🧬PyObject: The C Struct Behind Every Python ObjectINTERNALS

Every Python object is represented as a C struct that starts with the PyObject header: Py_ssize_t ob_refcnt (8 bytes) + PyTypeObject *ob_type (8 bytes). That's 16 bytes of overhead on every object, no matter how small. Then each type adds its own fields.

A Python int is PyLongObject: header + digit array. A Python list is PyListObject: header + ob_item (pointer to C array of PyObject*) + ob_size + allocated. This is why sys.getsizeof([])) is 56 bytes — 16 for header, 8 for ob_item ptr, 8 for size, 8 for allocated, + padding.

The type object (PyTypeObject) is a massive C struct with 50+ function pointer slots: tp_new, tp_init, tp_dealloc, tp_repr, tp_str, tp_hash, tp_call, tp_getattro, tp_setattro, tp_richcompare, number protocol (nb_add, nb_multiply...), sequence protocol (sq_item, sq_length...), etc. When you define a dunder method in Python, CPython fills in the corresponding slot.

CPython C STRUCT LAYOUT ────────────────────────────────────────────────────────────── PyObject (base header — every object starts with this): ┌─────────────────────┬─────────────────────┐ │ ob_refcnt (8 bytes)│ *ob_type (8 bytes)│ └─────────────────────┴─────────────────────┘ PyLongObject (int 42): ┌──────────┬──────────┬──────────┬──────────┐ │ ob_refcnt│ *ob_type │ob_digit[0]│ ... │ └──────────┴──────────┴──────────┴──────────┘ sys.getsizeof(0) = 24 bytes sys.getsizeof(2**100) = 40 bytes (more digits) PyListObject (list): ┌──────────┬──────────┬──────────┬──────────┬──────────┐ │ ob_refcnt│ *ob_type │ ob_size │ *ob_item │allocated │ └──────────┴──────────┴──────────┴──────────┴──────────┘ │ ▼ (heap-allocated C array) ┌────┬────┬────┬────┐ │*p0 │*p1 │*p2 │NULL│ PyObject* pointers └────┴────┴────┴────┘ Elements are NOT stored inline — only pointers to other PyObjects This is why Python lists can hold mixed types but are cache-unfriendly
⚠️ Why Python lists are cache-unfriendly: A Python list stores an array of pointers to PyObjects. Each element is a separate heap allocation at a random memory address. Iterating a list of 1M ints involves 1M pointer dereferences to random locations — exactly what defeats CPU L1/L2 caches. NumPy arrays, by contrast, store values inline in a contiguous C buffer — sequential memory = cache-friendly = 100× faster iteration.
NumPy Internals · 05
ndarray: Memory Layout
An ndarray is a thin Python wrapper around a raw C buffer. The entire performance story of NumPy comes from how that buffer is organised and accessed.
🗃️The ndarray Struct: What's Actually in MemoryINTERNALS

Every numpy.ndarray is a Python object containing these fields: data pointer (address of the raw C buffer), dtype (the element type descriptor — not just "float64" but also byte order, itemsize, and kind), shape (tuple of dimension sizes), strides (tuple of byte offsets per dimension step), flags (C-contiguous, Fortran-contiguous, writeable, etc.), and base (the array this view refers to, if any).

The data buffer is a flat, contiguous block of bytes — a C malloc allocation. For a float64 array, each element is exactly 8 bytes, laid out one after another. There are no Python object headers per element — just raw bytes. A 1000×1000 float64 matrix is exactly 8,000,000 bytes (8 MB).

Strides are the key to understanding everything about NumPy — views, slices, transposes, broadcasting. The stride for dimension i is the number of bytes you must advance in the buffer to move one step in that dimension.

ndarray MEMORY STRUCTURE ────────────────────────────────────────────────────────────── ndarray object (Python heap): ┌────────────────────────────────────────────────────────┐ │ data ptr → 0x7f3a2b000000 (C malloc'd buffer) │ │ dtype → float64 (itemsize=8, kind='f', '>') │ │ shape → (3, 4) │ │ strides → (32, 8) ← bytes to step each dimension │ │ flags → C_CONTIGUOUS=True, WRITEABLE=True │ │ base → None ← owns the data buffer │ └────────────────────────────────────────────────────────┘ │ ▼ C malloc buffer (3×4 float64 = 96 bytes) offset: 0 8 16 24 32 40 48 56 64 72 80 88 [00][01][02][03][10][11][12][13][20][21][22][23] row0──────────────row1──────────────row2────────── ELEMENT ACCESS: a[i, j] = data_ptr + i*strides[0] + j*strides[1] a[1, 2] = 0x7f... + 1*32 + 2*8 = offset 48 → element [1,2] TRANSPOSE (.T) — NO DATA COPY, just swap strides: a.T.shape = (4, 3) a.T.strides = (8, 32) ← same buffer, strides swapped Cost: O(1) — just creates new Python object with new strides
👁️Views vs Copies: The Stride EngineGOTCHA

A view is an ndarray that shares the same underlying data buffer as another array. Only the metadata (shape, strides, data offset) differs. Creating a view is O(1) — no memory allocation, no data copy.

Slicing creates views when possible. a[1:3] creates a new ndarray pointing to the same buffer but with an offset data pointer and smaller shape. a[::2] creates a view with doubled strides. a.T swaps strides. None of these copy data.

When copies happen: Fancy indexing (a[[0,2,4]]) cannot be expressed as a stride change — it creates a copy. Boolean indexing creates a copy. Any operation that requires non-contiguous output creates a copy. You can check: np.shares_memory(a, b) returns True for views.

The write-through gotcha: Modifying a view modifies the original. This is the most common NumPy bug. Always .copy() if you need independence.

numpy_internals.py
import numpy as np a = np.arange(12, dtype=np.float64).reshape(3, 4) # Inspect the ndarray metadata print(a.shape) # (3, 4) print(a.strides) # (32, 8) ← 32 bytes per row, 8 per element print(a.dtype) # float64 print(a.itemsize) # 8 bytes per element print(a.nbytes) # 96 total bytes print(a.flags) # C_CONTIGUOUS, WRITEABLE, OWNDATA print(a.__array_interface__['data']) # (raw_ptr_address, False) # Transpose: O(1) stride swap, no copy t = a.T print(t.strides) # (8, 32) ← swapped print(np.shares_memory(a, t)) # True — same buffer # Slice creates a view (shifted pointer, smaller shape) row = a[1] # array([4., 5., 6., 7.]) print(row.base is a) # True — row is a view into a row[0] = 999 print(a[1, 0]) # 999.0 ← a was modified! # Fancy indexing — creates a copy fancy = a[[0, 2]] # selects rows 0 and 2 print(fancy.base is None) # True — has its own data buffer # np.ascontiguousarray: force C-contiguous layout col = a[:, 0] # column view — strides=(32,), NOT contiguous! print(col.flags.c_contiguous) # False col_c = np.ascontiguousarray(col) # copies to C-contiguous buffer print(col_c.strides) # (8,) ← now contiguous

NumPy Internals · 06
Vectorisation, SIMD & BLAS
NumPy's speed isn't magic — it's three layers of acceleration: no Python loop overhead, CPU SIMD instructions, and optimised BLAS libraries for linear algebra.
⚡Layer 1: Eliminating Python Loop OverheadPERFORMANCE

A Python for loop over a list is expensive: each iteration executes multiple bytecode instructions, performs type lookups (LOAD_FAST, BINARY_OP), and creates/destroys Python int objects. For 1M iterations, that's ~50M C instructions just for loop machinery.

NumPy's ufuncs (universal functions like np.add, np.multiply) are implemented as C functions. When you write a + b on two arrays, NumPy calls PyUFunc_GenericFunction, which iterates the C buffer in a tight C loop with no Python overhead per element. The Python interpreter is invoked exactly once, regardless of array size.

Ufunc dispatch: NumPy checks the dtypes of inputs against a type resolution table. For float64 + float64, it picks the double loop C function. For mixed types, it promotes according to type-casting rules before computing.

🖥️Layer 2: SIMD — Single Instruction, Multiple DataDEEP DIVE

Modern CPUs have SIMD instruction sets (SSE2, AVX, AVX-512) that can operate on multiple values simultaneously in a single instruction. An AVX2 register is 256 bits — it can hold and process 4 float64 values at once or 8 float32 values at once.

NumPy (since 1.17) uses its own SIMD dispatch system: at build time, multiple versions of each inner loop are compiled for different SIMD targets. At import time, it detects which instructions the CPU supports (via CPUID) and selects the best available implementation. On a modern AVX-512 machine, array addition can process 8 float64s per clock cycle.

Cache effects: SIMD works best on contiguous memory. Sequential reads from a C-contiguous array allow the CPU's hardware prefetcher to load data into L1 cache before it's needed. Non-contiguous access (like iterating a Fortran-order array row-by-row) breaks the prefetcher and causes cache misses — often 10–100× slower.

📐Layer 3: BLAS/LAPACK for Linear AlgebraPERFORMANCE

np.dot(A, B) for large matrices does not use NumPy's own loop. It calls BLAS (Basic Linear Algebra Subprograms) — specifically the DGEMM routine (double-precision general matrix multiply). Your NumPy installation links to a BLAS implementation: OpenBLAS, MKL (Intel), BLIS, or Apple's Accelerate.

These BLAS libraries use highly optimised algorithms: cache-aware blocking (breaking matrices into tiles that fit in L1/L2 cache), multi-threaded execution, and hand-written assembly SIMD code. OpenBLAS's DGEMM for a 1000×1000 matrix approaches the theoretical peak FLOP/s of the CPU.

Check what BLAS you have: np.show_config(). MKL (included with Anaconda) is generally fastest on Intel CPUs. For non-matrix-multiply operations, NumPy uses its own SIMD dispatch.

PERFORMANCE COMPARISON: PYTHON LOOP vs NumPy ────────────────────────────────────────────────────────────── Task: add 1 to each element of 1,000,000 floats Python list + loop: ~400 ms for i in range(n): result[i] = lst[i] + 1 ↳ 1M × [LOAD_FAST + BINARY_OP + STORE_FAST + object alloc] NumPy ufunc: ~0.7 ms (570× faster) a + 1 ↳ 1 Python call → C ufunc loop → AVX2: 4 adds per cycle np.dot(A, B) call stack: Python: np.dot(A, B) ↓ NumPy: check contiguity, promote dtypes ↓ CBLAS: cblas_dgemm(CblasRowMajor, CblasNoTrans, ...) ↓ OpenBLAS: cache-blocked DGEMM, multi-threaded, AVX-512 ↳ approaches ~16 GFLOP/s on modern CPU
NumPy Internals · 07
Broadcasting Internals
Broadcasting is implemented via virtual strides of 0. No data is ever copied. Understanding this makes broadcasting intuitive and eliminates errors.
📡How Broadcasting Works in CINTERNALS

When NumPy broadcasts, it doesn't allocate a larger array and copy the data. Instead, it creates a temporary view with a stride of 0 on the broadcast dimension. A stride of 0 means: every step in that dimension reads the same memory address — as if the data repeats.

For a + b where a.shape=(3,1) and b.shape=(1,4): NumPy creates virtual views where a becomes shape (3,4) with strides (8, 0) and b becomes shape (3,4) with strides (0, 8). The ufunc loop iterates indices (i,j) and accesses a.data + i*8 + j*0 and b.data + i*0 + j*8 — the zero stride makes the "expanded" dimension read the same values repeatedly. No copy, pure pointer arithmetic.

The broadcasting rules (formally): Shapes are aligned right-to-left. If sizes match, fine. If one is 1, it broadcasts. If neither matches and neither is 1, ValueError. The output shape is the element-wise max of each dimension.

BROADCASTING VIA ZERO STRIDES ────────────────────────────────────────────────────────────── a = np.array([[1],[2],[3]]) # shape (3,1), strides (8, 8) b = np.array([10,20,30,40]) # shape (4,), strides (8,) Step 1: align shapes right-to-left a: (3, 1) → broadcast dim 1 → (3, 4) b: ( 4) → prepend 1 → (1, 4) → broadcast dim 0 → (3, 4) Step 2: create virtual views with 0-strides a_virtual: shape=(3,4), strides=(8, 0) ← 0 stride on axis 1 a_virtual[i,j] = a.data + i*8 + j*0 (j never advances) b_virtual: shape=(3,4), strides=(0, 8) ← 0 stride on axis 0 b_virtual[i,j] = b.data + i*0 + j*8 (i never advances) Step 3: ufunc loop over (3,4) output out[0,0]=a[0,0]+b[0]=1+10=11 out[0,1]=1+20=21 ... out[1,0]=2+10=12 out[1,1]=2+20=22 ... out[2,0]=3+10=13 ... ┌──────────────────────────────┐ │ 11 21 31 41 │ │ 12 22 32 42 │ │ 13 23 33 43 │ └──────────────────────────────┘ No data copied — pure pointer arithmetic with 0-strides

import numpy as np # Create a zero-stride view manually (what broadcasting does internally) a = np.array([1.0, 2.0, 3.0]) # shape (3,), strides (8,) # Broadcast to (3, 4) without copying a_virtual = np.lib.stride_tricks.as_strided( a, shape=(4, 3), strides=(0, 8) # 0-stride on axis 0 ) print(a_virtual) # 4 identical rows: [1. 2. 3.] repeated print(np.shares_memory(a, a_virtual)) # True # np.broadcast_shapes — compute output shape without doing the operation np.broadcast_shapes((3,1), (1,4)) # (3, 4) np.broadcast_shapes((8,1,6), (7,1)) # (8, 7, 6) # Common broadcasting patterns X = np.random.randn(100, 5) # 100 samples, 5 features mu = X.mean(axis=0) # shape (5,) std = X.std(axis=0) # shape (5,) X_norm = (X - mu) / std # broadcasts (5,) → (100,5) via 0-strides # Outer product via broadcasting a = np.arange(4)[:, np.newaxis] # shape (4, 1) b = np.arange(4)[np.newaxis, :] # shape (1, 4) outer = a * b # shape (4, 4) — multiplication table
PyTorch Internals · 08
Tensor Storage & Strides
PyTorch tensors share the same stride-based memory model as NumPy, but add device abstraction (CPU/CUDA/MPS), dtype flexibility (float16, bfloat16), and the autograd metadata layer on top.
💾Storage Object: The Real Data ContainerINTERNALS

A PyTorch Tensor is a Python object that holds metadata (shape, strides, dtype, device, grad_fn) plus a reference to a Storage object. The Storage is a flat, typed C buffer — the actual data. Multiple tensors can share the same Storage with different views (different offsets, shapes, strides) — exactly like NumPy views.

tensor.data_ptr() returns the raw C pointer to the start of the tensor's data within the storage. tensor.storage() returns the underlying storage. tensor.storage_offset() is the element offset into the storage where this tensor starts.

On GPU: CUDA tensors have their Storage in GPU VRAM. Operations on CUDA tensors compile and dispatch CUDA kernels — C++ code that runs on thousands of GPU cores in parallel. The API is identical to CPU tensors, but the execution is entirely different.

bfloat16 vs float16: Both are 16-bit but with different bit allocation. float16: 5 exponent bits, 10 mantissa. bfloat16: 8 exponent bits (same as float32), 7 mantissa. bfloat16 has the same numerical range as float32 but lower precision — it avoids float16's common overflow/underflow in training gradients, which is why it's preferred for ML.

PyTorch Internals · 09
The Autograd Engine
Autograd is an automatic differentiation system that builds a computation graph as you do math, then traverses it in reverse to compute gradients. Here's the exact mechanism.
∂Dynamic Computation Graph: How It's BuiltINTERNALS

When you perform an operation on tensors where any input has requires_grad=True, PyTorch records the operation in a directed acyclic graph (DAG). Each node in the graph is a Function object (a grad_fn). The output tensor stores a reference to its grad_fn, which in turn holds references to its inputs' grad_fns.

The graph is built dynamically — during the forward pass, as operations execute. This is PyTorch's key design difference from TensorFlow 1.x (which built a static graph before any computation). Dynamic graphs mean you can use Python control flow (if/for/while) freely — the graph reflects the actual execution path, not a pre-compiled template.

Leaf tensors (created by the user, not as the output of an operation) don't have a grad_fn. They accumulate gradients in .grad. Non-leaf tensors (intermediate results) have grad_fn. By default, PyTorch only retains gradients for leaf tensors with requires_grad=True.

COMPUTATION GRAPH + BACKPROP ────────────────────────────────────────────────────────────── Forward pass: L = ((x * w) + b)² x (leaf, no grad) ──► w (leaf, grad) ──────► MulBackward ──► AddBackward ──► PowBackward ──► L (scalar) b (leaf, grad) ──────────────────► │ │ ↑ grad_fn chain ↑ L.grad_fn = PowBackward0 L.grad_fn.next_functions[0][0] = AddBackward0 AddBackward0.next_functions[0][0] = MulBackward0 MulBackward0.next_functions[0][0] = AccumulateGrad (w) Backward pass: L.backward() ────────────────────────────────────────────────────────────── Starting gradient: dL/dL = 1 PowBackward: dL/d(·) = 2*(xw+b)¹ * 1 AddBackward: dL/d(xw) = dL/d(·), dL/db = dL/d(·) MulBackward: dL/dw = dL/d(xw) * x dL/dx = dL/d(xw) * w (not accumulated, x has no grad) w.grad += dL/dw ← accumulated into leaf tensor's .grad b.grad += dL/db ← accumulated into leaf tensor's .grad
🔄Gradient Accumulation, zero_grad(), and GotchasGOTCHA

Gradients are accumulated in .grad — not replaced. Each .backward() call adds to existing gradients. This is intentional for gradient accumulation training (simulating larger batch sizes), but it means you must call optimizer.zero_grad() before each backward pass in a standard training loop. Forgetting this is one of the most common PyTorch bugs.

torch.no_grad() is a context manager that disables gradient tracking entirely. All operations inside won't record grad_fn, won't build the graph. Use for inference and validation — it's faster and uses less memory because no computation graph is constructed.

tensor.detach() creates a new tensor that shares the same storage but has no gradient history (grad_fn=None, requires_grad=False). Use to pass tensors to non-differentiable operations (like NumPy) or to break graph connections deliberately.

retain_graph: By default, the graph is freed after .backward(). Call loss.backward(retain_graph=True) if you need to call backward again on the same graph (e.g. in some GAN setups or second-order optimisation).

Async Internals · 10
Event Loop: epoll, I/O Multiplexing & Scheduling
The asyncio event loop is not magic — it's a thin Python wrapper around OS-level I/O multiplexing (epoll on Linux, kqueue on macOS, IOCP on Windows).
🔄How the Event Loop Actually WorksINTERNALS

The event loop runs a tight loop. Each iteration: (1) call all ready callbacks (scheduled via call_soon), (2) calculate the earliest timer expiry, (3) call epoll_wait(fd, events, timeout) — a single blocking syscall that waits for any registered file descriptor to become readable/writable, or until the next timer fires, (4) process all returned I/O events and mark the waiting coroutines as ready, (5) repeat.

epoll is a Linux kernel feature. You register file descriptors (sockets, files, pipes) with the kernel via epoll_ctl(). Then a single epoll_wait() call blocks until any of those FDs has data — the kernel tells you which ones are ready. This scales to tens of thousands of simultaneous connections in one thread (the C10K solution).

The key insight: The event loop thread never blocks waiting for one specific connection. It hands all "please wake me when X is ready" registrations to the kernel, then blocks in a single epoll_wait. The kernel does all the monitoring in kernel space. Your code only runs when there's actually something to do.

ASYNCIO EVENT LOOP: ONE ITERATION ────────────────────────────────────────────────────────────── while True: ① Run ready callbacks (from _ready deque) ├─ Coroutine A resumed (was awaiting sleep) ├─ Coroutine B resumed (socket became readable) └─ Handle C (call_soon scheduled callback) ② Compute timeout (soonest timer - now) soonest timer: 0.023s from now ③ epoll_wait(epoll_fd, events, timeout=0.023) ↓ kernel blocks this thread for up to 23ms ↓ kernel returns when: a socket has data OR timer expires returns: [(socket_42, EPOLLIN), (socket_17, EPOLLOUT)] ④ Process I/O events socket_42 readable → resume awaiting coroutine X socket_17 writable → resume awaiting coroutine Y ⑤ Process expired timers asyncio.sleep(0.02) expired → resume coroutine Z CALL STACK WHEN COROUTINE RUNS ────────────────────────────────────────────────────────────── _run_once() ← event loop iteration Handle.__call__() ← callback wrapper Task.__step() ← coroutine driver coroutine.send(None) ← resume coroutine user code... ← your async def runs here await something ← yields Future back to loop

Coroutine Objects & the await Protocol
async def functions compile to coroutine objects. await is syntactic sugar over a generator-based protocol that dates back to PEP 342.
🔌What a Coroutine Object Actually IsINTERNALS

An async def function compiles to a regular Python function that, when called, returns a coroutine object. This object is a Python object implementing the coroutine protocol: it has send(), throw(), and close() methods. It does not execute any code when created.

The coroutine object holds a suspended execution frame — the same PyFrameObject described earlier, but frozen mid-execution. When send(value) is called, execution resumes from the last yield point (because at the bytecode level, await compiles to GET_AWAITABLE + YIELD_FROM).

The await chain: When you await some_coroutine, the current coroutine suspends and yields a Future object up through all the awaiting frames, all the way back to the event loop's Task.__step(). The Task registers a callback on the Future: "when this Future is done, call task.step() again." The event loop then runs other ready tasks until the Future resolves.

await PROTOCOL: CONTROL FLOW ────────────────────────────────────────────────────────────── async def my_coro(): result = await asyncio.sleep(1.0) # what happens here? Bytecode compiled as: GET_AWAITABLE ← calls sleep(1.0).__await__(), gets iterator YIELD_FROM ← suspends frame, passes Future up the chain EXECUTION TRACE ────────────────────────────────────────────────────────────── 1. Task.step() calls my_coro.send(None) 2. my_coro runs until 'await asyncio.sleep(1.0)' 3. asyncio.sleep creates a Future, schedules a callback in 1s 4. Future is yielded all the way back via YIELD_FROM chain 5. Task.step() receives the Future, adds done-callback: future.add_done_callback(task.__step) 6. Task.step() returns — coroutine is SUSPENDED 7. Event loop runs other tasks... 8. 1 second later: loop fires the timer callback 9. Future is resolved with None, done-callback fires 10. Task.__step() calls my_coro.send(None) again 11. my_coro resumes at the line AFTER 'await' WHY await ONLY WORKS INSIDE async def: 'await' compiles to YIELD_FROM. YIELD_FROM only works in a generator/coroutine frame. Calling it in a regular function raises SyntaxError — the bytecode has nowhere to yield to.

Attention Mechanism: The Math
Attention is a weighted average of value vectors, where the weights come from the similarity between queries and keys. Here's the full derivation and what each matrix actually represents.
🧮Scaled Dot-Product Attention: Step by StepINTERNALS

Given an input sequence of n tokens, each represented as a d_model-dimensional vector (stacked as matrix X of shape [n, d_model]), attention works as follows:

1. Project to Q, K, V: Three learnable weight matrices W_Q, W_K, W_V (each shape [d_model, d_k]) project each token into three roles: Query ("what am I looking for?"), Key ("what do I contain?"), Value ("what do I contribute?").

Q = X · W_Q, K = X · W_K, V = X · W_V — each shape [n, d_k].

2. Compute attention scores: scores = Q · Kᵀ / √d_k — shape [n, n]. Entry scores[i,j] is the dot product between query i and key j — a measure of how much token i should "attend to" token j.

Why divide by √d_k? Dot products grow with dimension. For large d_k, scores become very large, pushing softmax into regions with tiny gradients (the "softmax saturation" problem). Dividing by √d_k keeps the variance of the dot product ≈1, regardless of d_k.

3. Softmax: weights = softmax(scores, axis=-1) — shape [n, n]. Each row sums to 1. These are the attention weights: how much each token attends to each other token.

4. Aggregate values: output = weights · V — shape [n, d_k]. Each output token is a weighted average of all value vectors, where the weights came from query-key similarity.

🔀Multi-Head Attention: Parallel Attention SubspacesDEEP DIVE

A single attention head computes one kind of relationship. Multi-head attention runs h attention heads in parallel, each with its own Q/K/V projection matrices. Each head can specialise: one might learn syntactic relationships, another might learn coreference, another semantic roles.

With h heads and d_model dimensions, each head uses d_k = d_model / h dimensions. The total computation is similar to a single head with full dimensionality. After computing h outputs (each [n, d_k]), they're concatenated → [n, d_model] and projected by W_O back to [n, d_model].

Flash Attention (2022): Standard attention has O(n²) memory complexity — it materialises the full [n, n] score matrix. For n=4096, that's 4096² = 16M entries per head per layer. FlashAttention reformulates the computation to use tiling, computing attention in blocks that fit in GPU SRAM (L1 cache equivalent). Result: O(n) memory, same output, 2–4× faster for long sequences.

import torch import torch.nn.functional as F import math def scaled_dot_product_attention(Q, K, V, mask=None): """ Q: [batch, heads, seq_q, d_k] K: [batch, heads, seq_k, d_k] V: [batch, heads, seq_k, d_v] Returns: [batch, heads, seq_q, d_v], [batch, heads, seq_q, seq_k] """ d_k = Q.size(-1) # Scores: Q·Kᵀ / √d_k scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k) # scores shape: [batch, heads, seq_q, seq_k] # Causal mask (decoder: prevent attending to future tokens) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) # -inf → softmax → 0.0 attention weight # Softmax over key dimension attn_weights = F.softmax(scores, dim=-1) # [batch,heads,seq_q,seq_k] # Weighted sum of values output = torch.matmul(attn_weights, V) # [batch,heads,seq_q,d_v] return output, attn_weights class MultiHeadAttention(torch.nn.Module): def __init__(self, d_model, n_heads): super().__init__() assert d_model % n_heads == 0 self.n_heads = n_heads self.d_k = d_model // n_heads # Single matrix for Q, K, V projections (fused for efficiency) self.W_qkv = torch.nn.Linear(d_model, 3 * d_model, bias=False) self.W_o = torch.nn.Linear(d_model, d_model) def forward(self, x, mask=None): B, N, D = x.shape # Project to Q, K, V all at once (3D split) qkv = self.W_qkv(x).reshape(B, N, 3, self.n_heads, self.d_k) qkv = qkv.permute(2, 0, 3, 1, 4) # [3, B, heads, N, d_k] Q, K, V = qkv.unbind(0) out, weights = scaled_dot_product_attention(Q, K, V, mask) # Concatenate heads: [B, heads, N, d_k] → [B, N, d_model] out = out.transpose(1, 2).contiguous().reshape(B, N, D) return self.W_o(out), weights

Tokenisation: BPE & WordPiece
Tokenisation converts raw text to integer IDs. The algorithm used — BPE, WordPiece, or SentencePiece — deeply affects model capability, vocabulary coverage, and multilingual behaviour.
✂️Byte-Pair Encoding (BPE): The AlgorithmINTERNALS

BPE starts at the character level and iteratively merges the most frequent pair of adjacent symbols. Start with each character as a token. Count all adjacent pairs in the training corpus. Merge the most frequent pair into a single new token. Repeat until vocabulary size is reached.

Example: corpus contains "low" and "lower" many times. After many iterations, "low" becomes a single token. "lower" becomes "low" + "er". "lowest" becomes "low" + "est". Words the model has never seen are decomposed into known subwords, with the worst case falling back to individual characters — the model never encounters a true "unknown" token.

GPT-2/GPT-3/GPT-4 use BPE at the byte level (Byte-level BPE). All possible bytes (256) are the initial vocabulary, so every possible input string can be encoded, regardless of language or special characters. BERT uses WordPiece (similar idea but uses likelihood instead of frequency for merges).

Embeddings & SentenceTransformer Internals
Sentence embeddings are the output of a specific pooling strategy applied to the transformer's token-level outputs. The choice of pooling and the training objective define what the embedding space represents.
📐From Tokens to Sentence Vectors: What SentenceTransformer DoesINTERNALS

A transformer like BERT produces one output vector per input token — for a 20-token sentence, BERT returns a [20, 768] matrix. To get a single sentence vector, you need a pooling strategy.

CLS token pooling: BERT inserts a special [CLS] token at position 0. The [CLS] output vector was meant to aggregate sequence-level information (it's used for classification in fine-tuned BERT). Using it as a sentence embedding sounds natural, but raw BERT's [CLS] vector is not trained to encode semantics — it works poorly for similarity tasks.

Mean pooling (SentenceTransformer default): Average all token output vectors, weighted by the attention mask (ignore padding). Result: [768] (or [384] for smaller models). Empirically much better than [CLS] for semantic similarity.

Why does mean pooling work? The transformer's self-attention layers allow every token to incorporate context from every other token. By the final layer, each token vector already encodes information about the full sentence. Averaging "mixes" these enriched representations into a single global vector.

SentenceTransformer PIPELINE ────────────────────────────────────────────────────────────── Input: "The cat sat on the mat" │ ▼ Tokenizer (WordPiece / BPE) [CLS] the cat sat on the mat [SEP] → 8 token IDs + attention mask │ ▼ Token Embedding Layer [vocab_size × d_model] Look up each ID in embedding table → [8 × 384] float32 matrix + Positional Encoding (learned or sinusoidal) │ ▼ N Transformer Layers (Self-Attention + FFN + LayerNorm) [8 × 384] → [8 × 384] → ... → [8 × 384] ← one vec per token each token now incorporates full context via attention │ ▼ Pooling Layer (mean pool over non-padding tokens) sum(token_vecs * attention_mask) / sum(attention_mask) → [384] float32 vector ← the sentence embedding │ ▼ (Optional) L2 normalise v / ||v||₂ → unit vector, cosine sim = dot product │ ▼ numpy array shape (384,) WHY COSINE SIMILARITY? Embeddings encode DIRECTION not magnitude. "cat" and "feline" → similar direction, similar magnitude "cat" (repeated 100×) → same direction, larger magnitude L2-normalising makes all vectors unit length → cosine similarity = dot product = magnitude-invariant
🏋️Training Objective: Why These Embeddings Capture SemanticsMENTAL MODEL

Off-the-shelf BERT embeddings are terrible for sentence similarity — they were trained with masked language modelling (predict masked tokens), not for semantic similarity. SentenceTransformer models are fine-tuned with contrastive objectives specifically designed to organise the embedding space semantically.

MultipleNegativesRankingLoss (MNRL): Given a batch of (anchor, positive) pairs, for each anchor, treat all other positives in the batch as negatives. Maximise similarity to the true positive while minimising similarity to all batch negatives. With batch size 64, each example has 63 negatives — efficient and scalable. Used by DPR, E5, BGE.

The geometry of the resulting space: Semantically similar sentences cluster together. The space has meaningful directions — v_king - v_man + v_woman ≈ v_queen style arithmetic often works at the sentence level. Nearest-neighbour search in this space is semantic search.

💡 Practical consequence: You cannot use any transformer model as a sentence encoder out-of-the-box — only models explicitly fine-tuned for sentence similarity work well. The sentence-transformers library maintains a leaderboard of models with benchmarks. For production retrieval: BAAI/bge-large-en-v1.5 or text-embedding-3-large (OpenAI API). For multilingual: paraphrase-multilingual-mpnet-base-v2.
