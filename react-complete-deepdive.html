01Foundation

JavaScript is Single-Threaded

Every function, every state update, every React render — one call stack, no parallelism. Any work React does on that thread blocks everything: input, animations, layout, paint.

This isn't an implementation detail — it's the central constraint. Every React performance optimization you'll ever make is an answer to: how do I put less work on this one thread?

Event loop — one thread, everything queued
JS Thread
User click
React renders
DOM commit
idle
UI / Paint
⛔ BLOCKED — waits for JS thread to finish
Input
⛔ BLOCKED — events queue behind current task
The direct implication:

If React re-renders 40 components unnecessarily because of one state change, that's 40 function calls + 40 virtual DOM diffs + 40 DOM comparisons — all synchronously on that thread. Unnecessary re-renders are real CPU work that delays everything else.

02React Internals

The Fiber Reconciler

React doesn't touch the real DOM directly. It maintains a Fiber tree — an internal linked-list where each node (Fiber) represents one component. When state changes, React processes this tree first, then commits to the DOM in one atomic operation.

Mental model — Fiber is a to-do list:

Each Fiber node is a unit of work. React walks the tree depth-first, processes each node, marks dirty ones, skips clean ones. With React 18's Concurrent Mode, React can pause mid-walk, yield the thread to the browser for input, then resume. This is possible because each node stores its own state.

What a Fiber node contains

// Simplified — what React internally stores for every component
{
  type:          MyComponent,   // function or class
  key:           'item-3',        // for list reconciliation

  // Tree links (linked list, not an array)
  child:         FiberNode,        // first child
  sibling:       FiberNode,        // next sibling
  return:        FiberNode,        // parent

  // State
  memoizedState: HookState,        // your useState / useReducer values
  memoizedProps: Props,            // current props
  pendingProps:  Props,            // incoming props (during render)
  flags:         WorkFlags,        // e.g. Update | Deletion | Placement

  // Double buffer
  alternate:     FiberNode,        // the "other" tree being built
}

Double buffering — current vs work-in-progress

React maintains two Fiber trees simultaneously: the current tree (on screen) and the work-in-progress tree (being built). When work finishes, React swaps them atomically — no partial UI states ever reach the screen.

The reconciliation walk — why memo skips subtrees

React does a depth-first walk. For each node: did type change? Did props change? If a component is wrapped in React.memo and props are shallowly equal — skip it and its entire subtree. That's the real win from memo: not just one component, but every child under it.

Why this matters:

The render phase is pure JS on the single thread. The commit phase touches the DOM (expensive). Your job is to keep the render phase small — fewer Fiber nodes visited per state change. Every optimization (memo, useMemo, useSelector, context splitting) reduces how many nodes React must re-evaluate in the walk.

03Basics

State & the Prop Drilling Problem

When multiple components need the same data, you "lift state up" to a common ancestor and pass it down as props. At scale, this becomes prop drilling: middle components that don't use the data still must accept and forward it.

// Every middle layer must accept + forward props it doesn't care about
function App() {
  const [user, setUser] = useState(null);
  return <Layout user={user} setUser={setUser} />;
}
function Layout({ user, setUser }) { // doesn't USE user, just passes it
  return <Main user={user} setUser={setUser} />;
}
function Main({ user, setUser }) {   // doesn't USE user, just passes it
  return <UserMenu user={user} />;
}
function UserMenu({ user }) {         // finally uses it — 3 levels deep
  return <span>{user?.name}</span>;
}

Both Context and Redux solve this: any component anywhere can read global state directly. The difference is how they trigger re-renders and how precisely you can target them.

04Context API

Context Internals

When the Provider's value prop changes, React does an Object.is(prevValue, nextValue) check. If they differ, React walks the Fiber tree looking for all nodes that called useContext(ThisContext) and marks every one of them dirty — regardless of which keys they actually used.

Context update propagation
Provider value changes
→
Object.is(prev, next) → false
→
Walk entire Fiber tree
Find all useContext(ThisCtx) nodes
→
Mark ALL dirty
→
Re-render ALL consumers
⚠️ React has no key-level subscriptions in Context. No filtering. It's a broadcast.
The fundamental limit:

Context does not track which keys you destructured from the value. If your context holds { user, cartCount, theme } and cartCount changes, every component that called useContext(AppContext) re-renders — even those only reading user.

The Object.is trap

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [cart, setCart] = useState({ count: 0 });

  // ❌ NEW object literal on every render of AppProvider
  // Even if user + cart didn't change, this is a new reference
  // Object.is({ user, cart }, { user, cart }) === false  ← always
  return <Ctx.Provider value={{ user, cart, setUser, setCart }}>{children}</Ctx.Provider>;
}

// ✅ useMemo stabilizes — only new object when deps actually change
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [cart, setCart] = useState({ count: 0 });

  const value = useMemo(
    () => ({ user, cart, setUser, setCart }),
    [user, cart]  // setUser/setCart are stable — don't need to be here
  );
  return <Ctx.Provider value={value}>{children}</Ctx.Provider>;
}

Context — Full Usage Code

// context/UserContext.jsx — the correct full pattern
import { createContext, useContext, useState, useMemo, useCallback } from 'react';

const UserContext = createContext(null);

// Always a custom hook — never call useContext(UserContext) directly in components
export function useUser() {
  const ctx = useContext(UserContext);
  if (!ctx) throw new Error('useUser must be inside <UserProvider>');
  return ctx;
}

export function UserProvider({ children }) {
  const [user, setUser]       = useState(null);
  const [loading, setLoading] = useState(false);

  const login = useCallback(async (email, pw) => {
    setLoading(true);
    const data = await loginAPI(email, pw);
    setUser(data.user); setLoading(false);
  }, []);

  const logout = useCallback(() => setUser(null), []);

  const value = useMemo(
    () => ({ user, loading, login, logout }),
    [user, loading, login, logout]
  );

  return <UserContext.Provider value={value}>{children}</UserContext.Provider>;
}

07Context API

Fixing Context Performance

The only structural fix: split contexts. One context per concern. Components subscribe only to what they need.

const UserCtx   = createContext();
const CartCtx   = createContext();
const OrdersCtx = createContext();

function AppProvider({ children }) {
  const [user]   = useState(null);
  const [cart]   = useState({ count: 0 });
  const [orders] = useState({ badge: 0 });
  return (
    <UserCtx.Provider value={user}>
      <CartCtx.Provider value={cart}>
        <OrdersCtx.Provider value={orders}>{children}</OrdersCtx.Provider>
      </CartCtx.Provider>
    </UserCtx.Provider>
  );
}

// ✓ Header subscribes only to UserCtx — won't re-render on cart changes
function Header() {
  const user = useContext(UserCtx);
  return <h1>{user?.name}</h1>;
}
// ✓ CartIcon subscribes only to CartCtx — isolated
function CartIcon() {
  const cart = useContext(CartCtx);
  return <span>{cart.count}</span>;
}

08Redux Toolkit

Redux — Concepts First

Redux is a separate library (not built into React). One global store, strict mutation rules, and selector-based subscriptions — each component subscribes to exactly the piece of state it needs.

Store
Single source of truth

One JS object. Wrapped in <Provider>. Lives outside React — state doesn't reset on re-renders.

Slice
Domain of state + its logic

createSlice bundles initialState, reducers, and auto-generates action creators. One slice per domain.

Action
Description of what happened

{ type: 'cart/addItem', payload: item }. You dispatch actions — never mutate state directly.

Reducer
Pure (state, action) → newState

RTK uses Immer internally — write "mutating" code, get immutable updates automatically.

useSelector
Surgical subscription to one slice

Creates an independent subscription. React-Redux runs === on the selected value after every dispatch. Re-render only if it changed.

useDispatch
Trigger a state change

Returns the dispatch function. Stable reference — safe in useCallback deps.

// store/cartSlice.js
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], count: 0 },
  reducers: {
    addItem:    (state, { payload }) => { state.items.push(payload); state.count++; },
    removeItem: (state, { payload }) => {
      state.items = state.items.filter(i => i.id !== payload);
      state.count--;
    },
  },
});
export const { addItem, removeItem } = cartSlice.actions;
export default cartSlice.reducer;

11Memoization

The Memoization Triangle

React.memo, useMemo, and useCallback are almost always misused in isolation. They only work as a system — all three together, or the whole thing collapses.

React.memo
Skip re-rendering a component
Wraps a component. If props are shallowly equal to previous render, React skips running the function and reuses last output — including entire subtree.
Fails if: parent passes inline objects or functions — new reference every render, shallow equality breaks.
useCallback
Stable function reference
Memoizes a function so the same reference is returned across renders (while deps unchanged). Prevents memo'd children seeing "new" function props.
Useless without: React.memo on the child. Stable references only matter if the child checks prop equality.
useMemo
Cache computed values
Memoizes a computed value. Used for (1) expensive calculations, (2) stabilizing object/array references passed as props to memo'd children.
Useless without: React.memo on the child for case 2. For expensive calculations, standalone useMemo is valid.
The pattern that works:

1. Wrap child in React.memo. 2. Wrap any function you pass it in useCallback. 3. Wrap any object/array you pass it in useMemo. Miss any one step and the whole thing collapses.

// The triangle in action
const CartIcon = memo(function CartIcon({ count, onOpen }) {
  return <button onClick={onOpen}>🛒 {count}</button>; // skips if count+onOpen unchanged
});

function Header() {
  const [user]  = useState({ name: 'Alice' });
  const [count] = useState(0);

  const handleOpen = useCallback(() => console.log('cart opened'), []);
  // useCallback: stable fn ref → CartIcon doesn't see "new" onOpen on every Header render

  return (
    <header>
      <span>{user.name}</span>
      <CartIcon count={count} onOpen={handleOpen} />
      // CartIcon skipped when user.name changes — count and onOpen are same ✓
    </header>
  );
}

12Memoization

useMemo — Deep Dive

Stores the return value of a function. Runs the function only when dependencies change; otherwise returns the cached result.

function OrderStats({ orders, filter }) {
  // Use case 1: expensive computation
  const stats = useMemo(() => ({
    total:   orders.reduce((s, o) => s + o.amount, 0),
    highest: Math.max(...orders.map(o => o.amount)),
  }), [orders]);

  // Use case 2: stable array ref → prevents memo'd child re-renders
  const filtered = useMemo(
    () => orders.filter(o => o.status === filter),
    [orders, filter]
  );

  return <OrderList items={filtered} />; // OrderList is memo'd — stable ref = skip
}

useMemo.jsx
Don't wrap everything:

useMemo has its own cost — memory + comparison on every render. Only use it for genuinely expensive calculations (sorting/filtering large arrays) or stable references for memo'd child props. Simple a + b doesn't need it.

13Memoization

useCallback — Deep Dive

Every render creates a new function object. useCallback memoizes the function so you get the same reference across renders (while deps unchanged).

function Parent() {
  const [count, setCount] = useState(0);

  // ❌ New function on every render → memo'd child always re-renders
  const handleBad = () => setCount(c => c + 1);

  // ✅ Same reference → memo'd child is skipped
  const handleGood = useCallback(() => setCount(c => c + 1), []);
  // Empty deps safe here — functional update avoids stale closure on count

  // With a dependency — recreated only when it changes
  const handleSubmit = useCallback(() => submitOrder(count), [count]);

  return <MemoChild onClick={handleGood} />;
}

useCallback.jsx
useCallback is just useMemo for functions:

useCallback(fn, deps) === useMemo(() => fn, deps). Use whichever reads more naturally in context.

14Memoization

React.memo — Deep Dive

Wraps a component. If all props are shallowly equal to the previous render, React skips running the function and reuses the last JSX output — including all children. That's the real win: skipping entire subtrees.

import { memo } from 'react';

// Default: shallow equality on all props
const ProductCard = memo(function ProductCard({ product, onAdd }) {
  return <div><h3>{product.name}</h3><button onClick={onAdd}>Add</button></div>;
});

// Custom comparison — true = skip re-render, false = do re-render
const ProductCard = memo(ProductCard, (prev, next) => {
  return prev.product.id === next.product.id
      && prev.product.price === next.product.price;
});

15React 18

Concurrent Rendering — React 18

Prior optimizations reduce how much work React does. Concurrent Mode addresses: what if the work is unavoidable — can we do it without blocking input?

Key idea — yielding the thread:

Concurrent Mode doesn't add parallelism. Still one thread. Instead, React's scheduler breaks rendering into small chunks and yields control back to the browser between them. User input can jump in between chunks. The render is interruptible.

16React 18

useTransition — Low-Priority Updates

Mark a state update as "low priority." React renders it in the background, pausing to handle urgent work (typing) first. UI stays responsive during expensive renders.

import { useState, useTransition } from 'react';

function SearchPage() {
  const [query,   setQuery]   = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = e => {
    setQuery(e.target.value);              // urgent — update input immediately

    startTransition(() => {               // non-urgent — can be interrupted
      setResults(bigList.filter(item =>
        item.name.toLowerCase().includes(e.target.value)
      ));
    });
  };

  return (
    <div>
      <input value={query} onChange={handleChange} />
      {isPending && <span>Updating...</span>}
      <ResultList items={results} />
    </div>
  );
}
// React may restart the transition render if new input arrives mid-render

useDeferredValue — Lagged Values

Like useTransition but for values you don't control — like a prop from a parent. Gives you a "lagged" version that updates in the background.

import { useDeferredValue, memo, useMemo } from 'react';

function SearchPage({ query }) {
  const deferredQuery = useDeferredValue(query);
  // deferredQuery lags behind query — React renders old value first,
  // then does a background render with new value
  return (
    <div>
      <input value={query} />            {/* live query — snappy */}
      <ResultList query={deferredQuery} /> {/* deferred — background */}
    </div>
  );
}

// ResultList MUST be memo'd — otherwise deferring has no effect
const ResultList = memo(function ResultList({ query }) {
  const results = useMemo(
    () => bigList.filter(item => item.name.includes(query)),
    [query]
  );
  return results.map(r => <div key={r.id}>{r.name}</div>);
});

Virtualization — DOM is the Bottleneck

Previous optimizations reduce JS work. Virtualization solves a different problem: DOM nodes are expensive. Each one has event listeners, layout boxes, style computation, accessibility entries, and GC cost.

Rendering 10,000 list items creates 10,000 DOM nodes — even if only 20 are visible. Virtualization keeps the DOM node count constant by recycling a small pool of nodes as you scroll.

10,000 nodes in memory + layout

recycled node (repositioned)
recycled node (repositioned)

constant DOM cost regardless of data size

State architecture is the most impactful:

The best optimization is avoiding unnecessary state centralization. If only two components share state, lift to their parent — don't put it in a global store. Less scope = fewer potential re-renders. A well-colocated component tree beats any amount of memo/useCallback on a poorly-structured one.

    <div>
      <input value={search} onChange={handleSearch} placeholder="search..." />
      {isPending && <span>Filtering...</span>}
      <FixedSizeList height={600} itemCount={filtered.length} itemSize={80} width="100%">
        {({ index, style }) => (
          <div style={style}><ProductCard product={filtered[index]} /></div>
        )}
      </FixedSizeList>
    </div>
  );
}
