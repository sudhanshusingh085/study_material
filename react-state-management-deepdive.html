Context vs Redux
Deep Dive

A ground-up explanation of React state management — from JavaScript's single thread to useMemo, useSelector, and when to reach for Redux Toolkit over Context. Written so you never need to ask again.

01 —JavaScript is Single-Threaded

Before we touch React at all, you must understand this: JavaScript has one thread. One worker. It can only do one thing at a time.

This is the foundation everything else sits on. React renders are synchronous operations on that single thread. If your render is slow, your UI freezes.

JS Thread
User clicks
→ React renders
→ DOM update
→ idle
Other work
BLOCKED — must wait for JS thread
Why this matters for Context & Redux:

Every time a component re-renders, React must run its function, diff the virtual DOM, and commit changes — all on that single thread. Unnecessary re-renders are the #1 React performance killer. Both Context and Redux can cause components to re-render. The difference is in how much control you have over WHEN that happens.

The Event Loop

JavaScript runs tasks one at a time from a queue. React hooks into this loop. When state changes, React schedules a re-render task. With React 18's Concurrent Mode, React can pause and resume renders — but it still runs on one thread. There is no parallelism.

setState called
→
React schedules task
→
Event loop picks it up
→
Component re-renders
→
DOM updated
This entire chain happens synchronously on the one JS thread

02 —What is State & Why Share It?

State is data that can change over time and causes your UI to re-render when it does. React components have local state via useState.

Local state is fine when only one component needs it. Problems start when multiple components need the same data.

// LOCAL state — fine, only this component cares
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

// SHARED state — now you have a problem
// Header needs: user name
// Sidebar needs: user avatar
// CartIcon needs: cart count
// OrderPage needs: user id to fetch orders
// Where does this state LIVE?

03 —The Prop Drilling Problem

The naive solution is to "lift state up" — move shared state to the nearest common ancestor, then pass it down via props. This is called prop drilling and it becomes a nightmare at scale.

// App.jsx — holds the user state
function App() {
  const [user, setUser] = useState(null);
  return <Layout user={user} setUser={setUser} />;
}

// Layout doesn't even USE user — just passes it down!
function Layout({ user, setUser }) {
  return (
    <div>
      <Header user={user} />        // passed down again
      <Sidebar user={user} />       // passed down again
      <Main user={user} setUser={setUser} />
    </div>
  );
}

// Every middle component must accept and forward props it doesn't need
// This is PROP DRILLING — painful, fragile, hard to refactor

prop-drilling.jsx
The Problem:

Middle components become "pass-throughs". They have no interest in user but must accept and forward it. If you rename the prop, you have to update every file in the chain. Add a new component anywhere in the tree and you must thread the prop through.

Solution: A global store that any component can read from directly — without props. This is what Context and Redux both solve.

04 —React Context — What It Is

Context is React's built-in solution. It lets you create a "broadcast" — a value that any component in the tree can subscribe to, without passing it through props.

Mental model:

Think of Context like a radio station. You broadcast on a frequency (the Provider). Any component anywhere in the tree can tune in (useContext) and get the current value. When the broadcast changes, every tuned-in component re-renders.

Context — Use When

  • Theme (dark/light mode)
  • User authentication state
  • Language / locale
  • Rarely-changing global config
  • Small-to-medium apps
  • Simple updates (replace whole object)

Redux — Use When

  • Complex state with many sub-values
  • Many unrelated components need same state
  • Frequent updates (cart, notifications)
  • Debugging matters (time-travel)
  • Large teams with shared state logic
  • Async actions (API calls → state)

05 —How Context Works (Internals)

Context is built into React. You create a context object, wrap components in a Provider, and consume it with useContext.

createContext()
→
creates a Context object
<Context.Provider value={...}>
→
broadcasts the value to all children
useContext(Context)
→
subscribes, gets the current value, re-renders on change
⚠️ The Critical Performance Trap:

When the Provider's value changes, EVERY component that called useContext(MyContext) re-renders — even if the specific piece of data it cares about didn't change. This is the biggest pain point with Context.

06 —Context — Full Working Code

Here's the complete pattern for using Context correctly. We'll build a user auth context as an example.

Step 1: Create the Context

// context/UserContext.js
import { createContext, useContext, useState } from 'react';

// 1. Create the context (default value is used when no Provider wraps it)
const UserContext = createContext(null);

// 2. Create a custom hook — this is the CORRECT pattern, not calling useContext directly
export function useUser() {
  const context = useContext(UserContext);
  if (!context) {
    throw new Error('useUser must be used inside <UserProvider>');
  }
  return context;
}

// 3. Create the Provider component — this holds the actual state
export function UserProvider({ children }) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(false);

  const login = async (email, password) => {
    setIsLoading(true);
    const data = await loginAPI(email, password);
    setUser(data.user);
    setIsLoading(false);
  };

  const logout = () => setUser(null);

  // This VALUE object is what consumers receive
  const value = { user, isLoading, login, logout };

  return (
    <UserContext.Provider value={value}>
      {children}
    </UserContext.Provider>
  );
}

07 —Context Performance Problem

Here's the catch. Imagine your context holds three things: user, cartIcon, and ordersIcon.

// One big context with multiple values
const AppContext = createContext();

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [cartIcon, setCartIcon] = useState({ count: 0 });
  const [ordersIcon, setOrdersIcon] = useState({ badge: 0 });

  // ⚠️ Every time ANY of these changes, ALL consumers re-render
  const value = { user, cartIcon, ordersIcon, setCartIcon, setOrdersIcon };

  return <AppContext.Provider value={value}>{children}</AppContext.Provider>;
}

// Header only cares about user
function Header() {
  const { user } = useContext(AppContext); // subscribes to WHOLE context
  // ⚠️ Re-renders even when only cartIcon changes!
  return <h1>{user?.name}</h1>;
}

performance-problem.jsx
The Problem in Your Words:

Yes — your instinct is right. If user, cartIcon, and ordersIcon are in the same context, and cartIcon changes (someone adds to cart), then Header will re-render even though it only shows the user name. That's wasted work on the single JS thread.

Solution 1: Split Contexts (Best for Context)

// Split into separate contexts — now each component only subscribes to what it needs
const UserContext = createContext();
const CartContext = createContext();
const OrdersContext = createContext();

function AppProvider({ children }) {
  return (
    <UserContext.Provider value={userValue}>
      <CartContext.Provider value={cartValue}>
        <OrdersContext.Provider value={ordersValue}>
          {children}
        </OrdersContext.Provider>
      </CartContext.Provider>
    </UserContext.Provider>
  );
}

// Header now ONLY subscribes to UserContext
function Header() {
  const { user } = useContext(UserContext); // ✓ won't re-render on cart changes
  return <h1>{user?.name}</h1>;
}

// CartIcon only subscribes to CartContext
function CartIcon() {
  const { cartIcon } = useContext(CartContext); // ✓ isolated
  return <span>{cartIcon.count}</span>;
}

08 —Redux Toolkit — What It Is

Redux is a separate library (not built into React) that provides a single global store for your app's state. It has a strict pattern for how state can change, which makes it predictable and debuggable.

Redux Toolkit (RTK) is the modern, official way to use Redux. It removes all the old boilerplate.

Mental model:

Redux is like a bank. Your state is the balance. The only way to change the balance is by submitting a "transaction" (action). There's a record of every transaction (Redux DevTools). Multiple people (components) can check the balance — but only through the teller window (useSelector). And each person can request only the data they need, not the whole database.

Core Redux Concepts

Store
The single source of truth

One JavaScript object containing all your app's state. Entire app gets it through a Provider.

Action
A description of what happened

A plain object like { type: 'user/setUser', payload: {...} }. You "dispatch" actions to trigger state changes.

Reducer
A function that handles state changes

Takes current state + action, returns new state. Must be pure (no side effects). RTK handles immutability for you.

Slice
RTK's way to group related state

One slice = one "section" of your store (userSlice, cartSlice, ordersSlice). Auto-generates actions + reducer.

useSelector
Read specific data from the store

The key to performance. Component subscribes ONLY to the specific piece it selects. Re-renders only when THAT piece changes.

useDispatch
Trigger a state change

Returns the dispatch function. You call it with an action to update the store.

09 —Slices & Store — The RTK Pattern

In RTK, you create "slices" — one per domain. Each slice has initial state, reducers, and auto-generates action creators.

How a Redux update flows:
Component calls dispatch(action)
→
Redux sends action to all reducers
→
Matching reducer updates state
Store holds new state
→
useSelector components re-render ONLY if their selected value changed

Step 2: Create a Slice

// store/userSlice.js
import { createSlice } from '@reduxjs/toolkit';

const userSlice = createSlice({
  name: 'user',                    // slice name (prefix for action types)
  initialState: {
    data: null,
    isLoading: false,
    error: null,
  },
  reducers: {
    // RTK uses Immer under the hood — you CAN mutate state here safely
    setUser: (state, action) => {
      state.data = action.payload;  // looks like mutation, but it's safe!
    },
    clearUser: (state) => {
      state.data = null;
    },
    setLoading: (state, action) => {
      state.isLoading = action.payload;
    },
  },
});

// createSlice auto-generates action creators with the same names as reducers
export const { setUser, clearUser, setLoading } = userSlice.actions;
export default userSlice.reducer;

Step 3: Create More Slices

// store/cartSlice.js
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], count: 0 },
  reducers: {
    addItem: (state, action) => {
      state.items.push(action.payload);
      state.count += 1;
    },
    removeItem: (state, action) => {
      state.items = state.items.filter(item => item.id !== action.payload);
      state.count -= 1;
    },
  },
});

export const { addItem, removeItem } = cartSlice.actions;
export default cartSlice.reducer;

11 —useSelector — Surgical Re-renders

This is where Redux beats Context on performance. useSelector lets each component subscribe to only the exact piece of state it needs.

// Header — only re-renders when state.user.data changes
import { useSelector, useDispatch } from 'react-redux';
import { clearUser } from './store/userSlice';

function Header() {
  // Selector function: takes entire state, returns the piece you want
  const user = useSelector(state => state.user.data);
  const dispatch = useDispatch();

  // ✓ Won't re-render when cart changes, only when state.user.data changes
  return (
    <header>
      <span>{user?.name}</span>
      <button onClick={() => dispatch(clearUser())}>Logout</button>
    </header>
  );
}

// CartIcon — only re-renders when state.cart.count changes
function CartIcon() {
  const count = useSelector(state => state.cart.count);
  // ✓ Won't re-render when user changes, only cart.count
  return <span className="badge">{count}</span>;
}

// OrdersIcon — only re-renders when orders.badge changes
function OrdersIcon() {
  const badge = useSelector(state => state.orders.badge);
  return <span>{badge}</span>;
}

components-with-redux.jsx
✓ Answer to your question:

Yes! This is exactly what useSelector does. Even though user, cartIcon, and ordersIcon are all in the same Redux store, Header won't re-render when cart changes because its selector state => state.user.data returned the same value. Redux does a strict equality check (===) on the selected value after each dispatch.

⚠️ useSelector Performance Trap — Returning Objects

// ❌ BAD — creates a NEW object on every call → always "changed" → always re-renders
const { name, email } = useSelector(state => ({
  name: state.user.data?.name,
  email: state.user.data?.email,
}));
// Even if name and email are the same, {} !== {} in JavaScript
// So this component re-renders on EVERY dispatch, everywhere

// ✅ GOOD option 1 — select primitives (strings, numbers, booleans)
const name = useSelector(state => state.user.data?.name);
const email = useSelector(state => state.user.data?.email);

// ✅ GOOD option 2 — use shallowEqual for object comparison (see next section)
import { shallowEqual } from 'react-redux';
const { name, email } = useSelector(
  state => ({ name: state.user.data?.name, email: state.user.data?.email }),
  shallowEqual   // second argument: custom comparison function
);

12 —useMemo — Cache Expensive Calculations

useMemo is about computation. It caches the result of a function. If the dependencies haven't changed, it returns the cached result instead of recomputing.

Mental model:

useMemo is like a sticky note. "Last time I ran this calculation with these inputs, I got this answer. If the inputs haven't changed, use the sticky note instead of recalculating."

import { useMemo } from 'react';

function OrderStats({ orders }) {
  // ❌ Without useMemo — recalculates on EVERY render
  const totalBad = orders.reduce((sum, o) => sum + o.amount, 0);

  // ✅ With useMemo — only recalculates when `orders` actually changes
  const total = useMemo(() => {
    console.log('Recalculating total...'); // only runs when orders changes
    return orders.reduce((sum, o) => sum + o.amount, 0);
  }, [orders]); // dependency array — recalculate when these change

  // Also useful for stable object references
  const sortedOrders = useMemo(() => {
    return [...orders].sort((a, b) => b.date - a.date);
  }, [orders]);

  return <div>Total: ${total} | Orders: {sortedOrders.length}</div>;
}

useMemo.jsx
When NOT to use useMemo:

Don't wrap everything in useMemo. It has its own overhead (memory + the check itself). Only use it for: (1) expensive calculations like sorting/filtering large arrays, (2) creating objects/arrays that are passed to child components as props (to prevent unnecessary child re-renders).

useMemo with Context to prevent re-renders

function UserProvider({ children }) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(false);

  // ❌ Without useMemo — new object on every render → all consumers re-render
  const valueBad = { user, isLoading, setUser, setIsLoading };

  // ✅ With useMemo — same object reference if user and isLoading didn't change
  const value = useMemo(() => ({
    user, isLoading, setUser, setIsLoading
  }), [user, isLoading]); // only recompute when these change

  return <UserContext.Provider value={value}>{children}</UserContext.Provider>;
}

13 —useCallback — Stable Function References

useCallback is like useMemo but for functions. It gives you a stable function reference across renders. Why does that matter?

// In JavaScript, functions are objects. Every render creates a NEW function.
function Parent() {
  const [count, setCount] = useState(0);

  // ❌ New function created on every render
  const handleClick = () => console.log('clicked');
  
  // This means Child sees a "new" prop every render → Child re-renders unnecessarily
  return <Child onClick={handleClick} />;
}

// ✅ useCallback memoizes the function — same reference if deps unchanged
function Parent() {
  const [count, setCount] = useState(0);

  const handleClick = useCallback(() => {
    console.log('clicked');
  }, []); // empty array = never recreate (safe if no dependencies)

  // Example with dependency
  const handleAdd = useCallback(() => {
    setCount(c => c + 1); // using updater fn avoids needing count in deps
  }, []); // still safe — setCount is stable from React

  return <Child onClick={handleClick} onAdd={handleAdd} />;
}

useCallback.jsx
useCallback vs useMemo:

useCallback(fn, deps) is exactly the same as useMemo(() => fn, deps). useCallback just reads more naturally when you're memoizing a function. Use useCallback for functions, useMemo for values.

14 —React.memo — Skip Re-rendering Child Components

React.memo is a higher-order component (a wrapper). If a component's props haven't changed, React.memo skips re-rendering it entirely.

import { memo, useCallback } from 'react';

// Wrap the component you want to protect from unnecessary re-renders
const CartIcon = memo(function CartIcon({ count, onClick }) {
  console.log('CartIcon rendered'); // only if count or onClick changed
  return <button onClick={onClick}>🛒 {count}</button>;
});

function Header() {
  const [user, setUser] = useState({ name: 'Alice' });
  const [cartCount, setCartCount] = useState(0);

  // ✅ useCallback ensures CartIcon doesn't re-render when user changes
  const handleCartClick = useCallback(() => {
    console.log('cart opened');
  }, []);

  return (
    <header>
      <span>{user.name}</span>
      {/* CartIcon won't re-render when user.name changes — count and onClick are same */}
      <CartIcon count={cartCount} onClick={handleCartClick} />
    </header>
  );
}

react-memo.jsx
React.memo requires stable props to work:

React.memo does a shallow equality check on props. If you pass an object or function as a prop without useMemo/useCallback, it gets a new reference every render, and React.memo has no effect. Always pair React.memo with useCallback/useMemo on the parent side.

15 —useSelector + shallowEqual Deep Dive

Back to your original question about Redux selectors. shallowEqual from react-redux compares two objects property-by-property rather than by reference.

import { useSelector, shallowEqual } from 'react-redux';
import { createSelector } from '@reduxjs/toolkit';

// APPROACH 1: Select primitives separately (simplest, most performant)
function UserProfile() {
  const name = useSelector(state => state.user.data?.name);
  const email = useSelector(state => state.user.data?.email);
  // Each is a primitive — strict equality works perfectly
}

// APPROACH 2: shallowEqual — compare object properties, not reference
function UserProfile() {
  const { name, email } = useSelector(
    state => ({ name: state.user.data?.name, email: state.user.data?.email }),
    shallowEqual  // checks: did any of name/email actually change?
  );
  // Won't re-render if name and email are the same, even though object reference is new
}

// APPROACH 3: createSelector — memoized selectors (best for complex derivations)
// Creates a selector that only recomputes when inputs change
const selectExpensiveOrders = createSelector(
  state => state.orders.list,        // input selector
  orders => orders.filter(o => o.amount > 1000)  // result is memoized
);

function ExpensiveOrdersList() {
  // Won't recompute filter unless orders.list changes
  const expensiveOrders = useSelector(selectExpensiveOrders);
  return expensiveOrders.map(o => <div key={o.id}>{o.amount}</div>);
}

16 —When to Use What — Decision Guide

Aspect Context Redux Toolkit
Bundle size 0kb — built into React ~10kb (RTK + react-redux)
Setup complexity Simple — 2 files More setup — slices, store, provider
Performance control Limited — split contexts or useMemo Precise — useSelector selects exactly what you need
DevTools React DevTools (basic) Redux DevTools — time travel, action log
Async (API calls) Manual with useEffect createAsyncThunk — built-in
Re-render control Whole context re-renders all subscribers Per-component via useSelector + shallowEqual
Learning curve Low — just React Medium — new concepts (slice, dispatch, selector)

The Decision Tree

Is this state used by more than one component?

Keep going ↓
Does it change frequently? (cart, notifications, live data)

Consider Redux for performance control
Do you have complex async flows or a large team?
Context with split providers works

17 —Full Example — Context + Redux Together

In real apps you often use both. Context for rarely-changing global settings. Redux for frequently-updating feature state.

// Real-world pattern: Theme in Context (never changes fast), everything else in Redux

// ThemeContext — changes rarely (user preference)
const ThemeContext = createContext('dark');
export const useTheme = () => useContext(ThemeContext);

// Redux store — everything that changes frequently
// state.user — login/logout
// state.cart — add/remove items frequently
// state.notifications — real-time badges

// main.jsx
ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>          {/* Redux for feature state */}
    <ThemeContext.Provider value="dark"> {/* Context for config */}
      <App />
    </ThemeContext.Provider>
  </Provider>
);

// Component using both
function ProductCard({ product }) {
  const theme = useTheme();                    // from Context
  const user = useSelector(s => s.user.data); // from Redux
  const dispatch = useDispatch();

  const handleAddToCart = useCallback(() => {
    dispatch(addItem({ id: product.id, name: product.name, price: product.price }));
  }, [dispatch, product]);

  return (
    <div className={`card card--${theme}`}>
      <h3>{product.name}</h3>
      {user ? (
        <button onClick={handleAddToCart}>Add to Cart</button>
      ) : (
        <button>Login to buy</button>
      )}
    </div>
  );
}

Summary of All Performance Tools

useMemo
Cache expensive computed values

Use when: filtering/sorting large arrays, deriving data, creating stable object references for child props

useCallback
Cache function references

Use when: passing functions as props to memoized children, functions in dependency arrays of other hooks

React.memo
Skip re-rendering a component

Use when: a child component renders often with the same props, and its render is expensive

useSelector
Subscribe to only part of Redux store

Use when: multiple components need the same state but different subsets of it

shallowEqual
Compare objects by properties, not reference

Use as second arg to useSelector when you must return an object and want stable comparison

createSelector
Memoized derived state from Redux

Use when: computing derived data from Redux state (filters, sorts, aggregations) that's shared across components

✓ The mental checklist before you code:

1. Does only one component need this? → local useState. 2. Do many components need it but it rarely changes? → Context. 3. Does it change often or need precise re-render control? → Redux Toolkit. 4. Is a computation expensive or passed as a prop? → useMemo. 5. Is a function passed as a prop to a memoized child? → useCallback. 6. Is a child component rendering too often with same props? → React.memo.
