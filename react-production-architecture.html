Folder Structure

Structure by feature, not by type. "All hooks in /hooks, all components in /components" breaks down at scale. Feature-based colocation keeps everything that changes together, together.

Production Structure

💡
The Barrel Rule: Every feature folder gets an index.ts that explicitly exports only what other features are allowed to use. If it's not in index.ts, it's private to that feature. This prevents spaghetti imports and enforces encapsulation.
❌ Don't — Type-based structure

import { useAuth } from '../../hooks/useAuth'
import { login } from '../../services/authService'
import { AuthState } from '../../types/auth'
✅ Do — Feature-based structure
// single import from feature barrel
import {
  LoginForm,
  useAuth,
  type AuthState
} from '@/features/auth'

Component Rules

Components should be dumb about data and smart about UI. The golden rule: a component that fetches data, holds state, AND renders JSX is doing too much.

RuleOne Component, One Responsibility
If your component does data fetching + business logic + rendering, split it. Container/Presenter pattern: container handles data, presenter handles UI.
// UserContainer.tsx — handles data
export function UserContainer() {
  const { data, isLoading } = useUser();
  return <UserCard user={data}
               loading={isLoading} />;
}
// UserCard.tsx — pure UI, no side effects
export function UserCard({ user, loading }) {
  if (loading) return <Skeleton />;
  return <div>{user.name}</div>;
}
DoCo-locate Component Files
A component's styles, tests, types and sub-components live next to it — not in separate top-level folders.
UserCard/ ├── UserCard.tsx ├── UserCard.test.tsx ├── UserCard.types.ts ├── UserAvatar.tsx // sub-component └── index.ts
Don'tNo Anonymous Default Exports
Anonymous exports break refactoring tools, make stack traces unreadable, and prevent fast refresh in dev.
❌ Bad
export default () => (
  <div>Hello</div>
);
✅ Good
export function Greeting() {
  return <div>Hello</div>;
}
RuleProps Interface Always Named
Name your props interface ComponentNameProps. Never use inline types for props — they can't be reused or documented.
// ✅ Correct
interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: 'primary' | 'ghost';
  disabled?: boolean;
}

export function Button({ label, onClick,
  variant = 'primary', disabled
}: ButtonProps) { ... }
NoteMax Props Rule
If a component takes more than 5–6 props, it's probably doing too much. Either split it or group related props into a config object.
// Too many props → use a data object
interface UserCardProps {
  user: User;             // group data
  actions: UserActions;  // group callbacks
  config?: CardConfig;   // group settings
}
PerfAvoid Prop Drilling > 2 Levels
If you're passing a prop through more than 2 components that don't use it, extract to Context or Zustand. Prop drilling is a maintenance hazard.

Hooks Architecture

Hooks are the backbone of React logic. The key architecture decision: where does a hook live? The answer depends entirely on how many places use it.

Hook Type Where to Put It Examples Rule
Component-localInline in component fileuseState, useEffectUsed in exactly 1 component
Feature hookfeatures/auth/useAuth.tsuseAuth, useCartUsed only within a feature
Shared hookhooks/useDebounce.tsuseDebounce, useLocalStorageUsed by 2+ features
API hookfeatures/users/useUsers.tsuseUsers, useUpdateUserWraps React Query calls
Store hookstore/uiStore.tsuseUIStore, useThemeAccesses global Zustand state
Essential Shared Hooks Catalog
useDebounce<T>
When to use → search inputs, autocomplete, resize handlers
Delays updating a value until the user stops changing it. Prevents spamming API calls on every keystroke.
useLocalStorage<T>
When to use → persisted preferences, last-used filters, theme
Synced useState that reads/writes to localStorage. SSR-safe with fallback to default value.
useClickOutside
When to use → dropdowns, modals, tooltips, popovers
Fires a callback when user clicks outside a referenced element. Attach via ref.
useMediaQuery
When to use → responsive conditional rendering
Returns a boolean for a CSS media query string. Replaces window.matchMedia boilerplate. SSR-safe.
usePrevious<T>
When to use → animate on change, diff old vs new value
Stores the previous render's value using a ref. One-liner but used everywhere.
useAsync
When to use → one-off async actions (not data fetching)
Wraps an async fn and tracks its loading/error/data state. For actions outside React Query.
useEventListener
When to use → keyboard shortcuts, scroll, custom DOM events
Attaches/removes event listeners safely with cleanup. Works on window, document, or any element ref.
useIntersectionObserver
When to use → lazy images, infinite scroll, scroll animations
Wraps IntersectionObserver API and returns whether an element is in viewport.
useTimeout / useInterval
When to use → polling, auto-dismiss toasts, countdowns
Declarative timers that auto-clean on unmount. Never write setInterval in useEffect again.
Hook Patterns — With Code
PatternuseDebounce
function useDebounce<T>(
  value: T,
  delay: number
): T {
  const [debounced, setDebounced] =
    useState<T>(value);

  useEffect(() => {
    const timer = setTimeout(
      () => setDebounced(value), delay
    );
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debounced;
}

// Usage:
const debouncedSearch = useDebounce(search, 400);
// query only fires after 400ms pause
PatternuseLocalStorage
function useLocalStorage<T>(
  key: string,
  initial: T
) {
  const [value, setValue] = useState<T>(() => {
    try {
      const item = window.localStorage
        .getItem(key);
      return item
        ? JSON.parse(item)
        : initial;
    } catch { return initial; }
  });

  const set = (val: T) => {
    setValue(val);
    localStorage.setItem(
      key, JSON.stringify(val)
    );
  };
  return [value, set] as const;
}
PatternuseClickOutside
function useClickOutside(
  ref: RefObject<HTMLElement>,
  handler: () => void
) {
  useEffect(() => {
    const listener = (e: MouseEvent) => {
      if (!ref.current ||
          ref.current.contains(
            e.target as Node)
      ) return;
      handler();
    };
    document.addEventListener(
      'mousedown', listener
    );
    return () =>
      document.removeEventListener(
        'mousedown', listener
      );
  }, [ref, handler]);
}
PatternFeature Data Hook (useUsers)
Wrap all React Query calls for a feature in one hook. Components never call useQuery directly.
// features/users/useUsers.ts
export function useUsers() {
  return useQuery({
    queryKey: ['users'],
    queryFn: fetchUsers,
    staleTime: 5 * 60 * 1000,
  });
}

export function useUpdateUser() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: updateUser,
    onSuccess: () =>
      qc.invalidateQueries({ queryKey: ['users'] }),
  });
}
🚫
Never do this in a hook: Don't put JSX in a hook. Don't call hooks conditionally. Don't create hooks that return components. A hook returns data/state/functions — never markup.

State

State Management

The #1 React mistake: putting everything in global state. The decision tree is simple — start local, go global only when needed.

🗺️
The State Decision Tree:
1. Does only one component need this? → useState
2. Do sibling components need it? → Lift to nearest parent
3. Do many components across the tree need it? → Zustand / Context
4. Does it come from the server? → React Query (never useState for this)
5. Does it need to survive page refresh? → localStorage / URL params
Zustand — Production Patterns
PatternSlice Pattern (one store per domain)
Don't put everything in one monster store. Create small focused stores per domain.
// store/uiStore.ts
interface UIStore {
  sidebarOpen: boolean;
  theme: 'light' | 'dark';
  toggleSidebar: () => void;
  setTheme: (t: UIStore['theme']) => void;
}

export const useUIStore =
  create<UIStore>()(set => ({
    sidebarOpen: false,
    theme: 'light',
    toggleSidebar: () =>
      set(s => ({ sidebarOpen: !s.sidebarOpen })),
    setTheme: theme => set({ theme }),
  }));
PerfSelect Only What You Need
Always use selectors. Subscribing to the whole store re-renders on any state change.
// ✅ Subscribes only to sidebarOpen
const isOpen = useUIStore(
  s => s.sidebarOpen
);

// ❌ Re-renders on ANY store change
const { sidebarOpen } = useUIStore();
PatternPersist to LocalStorage
import { persist } from 'zustand/middleware';

export const useSettingsStore =
  create()(persist(
    (set) => ({
      language: 'en',
      setLanguage: (lang) => set({ language: lang }),
    }),
    { name: 'app-settings' }
  ));
Don'tNever Store Derived Data
If a value can be computed from other state, compute it — don't store it. Stored derived data always gets out of sync.
// ❌ fullName is derived — don't store it
{ firstName: 'John', lastName: 'Doe',
  fullName: 'John Doe' }

// ✅ Compute it
const fullName = `${firstName} ${lastName}`
State ToolUse ForNot For
useStateLocal UI state (open/closed, input value)Server data, global UI state
useReducerComplex state with multiple sub-values or transitionsSimple boolean toggles
ZustandGlobal client UI state (sidebar, theme, user session)Server/async data
React QueryALL server data — fetching, caching, mutationsUI-only state
ContextRarely-changing config (theme, locale, auth user)Frequently changing data
URL / ParamsShareable state (filters, pagination, selected ID)Ephemeral UI state

Server State (React Query)

Server state is fundamentally different from client state — it's asynchronous, can be stale, and is shared across users. Never use useState for data that comes from an API. React Query handles this category entirely.

PatternQuery Key Factory
Centralize all query keys in one object. Prevents typos and makes invalidation trivial.
// lib/queryKeys.ts
export const queryKeys = {
  users: {
    all: ['users'] as const,
    list: (filters: Filters) =>
      ['users', 'list', filters] as const,
    detail: (id: string) =>
      ['users', 'detail', id] as const,
  },
};

// Usage:
useQuery({ queryKey: queryKeys.users.detail(userId) })
// Invalidate all user queries:
qc.invalidateQueries({ queryKey: queryKeys.users.all })
PatternOptimistic Updates
Update the UI immediately, roll back on error. Makes the app feel instant.
useMutation({
  mutationFn: toggleTodo,
  onMutate: async (id) => {
    await qc.cancelQueries({ queryKey: ['todos'] });
    const prev = qc.getQueryData(['todos']);
    qc.setQueryData(['todos'], old =>
      old.map(t => t.id === id
        ? { ...t, done: !t.done } : t)
    );
    return { prev }; // context for rollback
  },
  onError: (_, __, ctx) =>
    qc.setQueryData(['todos'], ctx?.prev),
  onSettled: () =>
    qc.invalidateQueries({ queryKey: ['todos'] }),
});
PerfstaleTime vs cacheTime
Set staleTime appropriately. Default is 0 (refetch on every focus). Most data doesn't change that fast.
// Rarely-changing reference data
useQuery({
  queryKey: ['countries'],
  queryFn: fetchCountries,
  staleTime: Infinity, // never refetch
});

// User-specific data
useQuery({
  queryKey: ['user', id],
  queryFn: () => fetchUser(id),
  staleTime: 5 * 60 * 1000, // 5 min
});
PatternInfinite Scroll / Pagination
const { data, fetchNextPage,
         hasNextPage, isFetchingNextPage }
  = useInfiniteQuery({
    queryKey: ['posts'],
    queryFn: ({ pageParam = 1 }) =>
      fetchPosts(pageParam),
    getNextPageParam: (lastPage) =>
      lastPage.hasMore
        ? lastPage.nextCursor
        : undefined,
  });

Forms — React Hook Form + Zod

The production standard: React Hook Form for form state (uncontrolled, performant) + Zod for schema validation. Never build form validation logic by hand at scale.

PatternFull Production Form Setup
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';

// 1. Define schema (reusable, shareable)
const loginSchema = z.object({
  email: z.string().email('Invalid email'),
  password: z.string().min(8, 'Min 8 characters'),
});
type LoginFormData = z.infer<typeof loginSchema>;

export function LoginForm() {
  const { register, handleSubmit, formState } =
    useForm<LoginFormData>({
      resolver: zodResolver(loginSchema),
      defaultValues: { email: '', password: '' },
    });

  const onSubmit = (data: LoginFormData) => {
    // data is fully type-safe and validated
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('email')} />
      {formState.errors.email &&
        <p>{formState.errors.email.message}</p>}
      <button disabled={formState.isSubmitting}>
        {formState.isSubmitting ? 'Logging in…' : 'Login'}
      </button>
    </form>
  );
}
PerfUse watch() Sparingly
watch() re-renders on every keystroke. Prefer getValues() in callbacks, or useWatch with a specific field name to limit re-renders.
// ✅ Only re-renders when 'email' changes
const email = useWatch({
  control, name: 'email'
});
PatternReusable Field Components
Wrap each field type into a reusable component that accepts name, label, control. Never repeat error/label logic.
function TextField({ name, label, control }) {
  return (
    <Controller name={name} control={control}
      render={({ field, fieldState }) => (
        <div>
          <label>{label}</label>
          <input {...field} />
          {fieldState.error &&
            <span>{fieldState.error.message}</span>}
        </div>
      )} />
  );
}
07
Performance

Rendering Optimizations

Optimize last, not first. Measure with React DevTools Profiler before reaching for memo/useMemo/useCallback. Premature optimization adds complexity without benefit.

⚠️
The Memo Trap: React.memo, useMemo, and useCallback all have overhead. They're only beneficial when (a) the component is actually re-rendering unnecessarily AND (b) the re-render is measurably expensive. Profile first.
PerfReact.memo — When to Use
Wrap a component with memo when it renders the same output given the same props and its parent re-renders frequently.
// ✅ Good candidate: heavy list item
// rendered 100x, parent updates often
const UserRow = React.memo(
  function UserRow({ user, onSelect }) {
    return <tr>...</tr>;
  },
  // Optional custom comparator
  (prev, next) => prev.user.id === next.user.id
);
PerfuseCallback — Stable References
Use useCallback only when passing callbacks to memoized children or as useEffect deps. Otherwise it's overhead with no benefit.
// ✅ Needed — onSelect passed to memo'd child
const handleSelect = useCallback(
  (id: string) => setSelected(id),
  [] // setSelected is stable
);

// ❌ Pointless — no memoized child using it
const handleClick = useCallback(
  () => console.log('click'), []
);
PerfuseMemo — Expensive Computations
Use useMemo for computationally expensive transforms, not for avoiding object reference equality issues.
// ✅ Expensive sort/filter over large array
const sortedUsers = useMemo(() =>
  [...users]
    .filter(u => u.active)
    .sort((a, b) =>
      a.name.localeCompare(b.name)
    ),
  [users]
);

// ❌ Not worth memoizing
const title = useMemo(
  () => `Hello ${name}`, [name]
);
PerfState Colocation
The single biggest performance win that requires no memoization: put state as close to where it's used as possible. State higher up = larger re-render subtrees.
// ✅ Input state lives in the input, not App
function SearchInput() {
  const [val, setVal] = useState('');
  return <input value={val}
           onChange={e => setVal(e.target.value)}/>;
}
PatternList Virtualization
For lists with 50+ items, use TanStack Virtual or react-window. Rendering 1000 DOM nodes is always slow — virtualization renders only visible rows.
import { useVirtualizer }
  from '@tanstack/react-virtual';

const virtualizer = useVirtualizer({
  count: items.length,
  getScrollElement: () => parentRef.current,
  estimateSize: () => 40, // row height in px
});
PatternAvoid Object/Array Literals in JSX
New objects/arrays created inline in JSX break shallow equality checks and cause unnecessary re-renders in memoized children.
// ❌ New array on every render
<Chart colors={['red', 'blue']} />

// ✅ Stable reference outside component
const CHART_COLORS = ['red', 'blue'];
<Chart colors={CHART_COLORS} />

Code Splitting & Lazy Loading

Ship only what the user needs right now. Every route should be a separate chunk. Heavy libraries (charts, rich text editors, maps) should be loaded on demand.

PatternRoute-Level Splitting (Standard)
// router.tsx — every page is lazy
const Dashboard = lazy(() =>
  import('@/pages/DashboardPage'));
const Settings = lazy(() =>
  import('@/pages/SettingsPage'));

<Suspense fallback={<PageSkeleton />}>
  <Routes>
    <Route path="/dashboard"
           element={<Dashboard />} />
    <Route path="/settings"
           element={<Settings />} />
  </Routes>
</Suspense>
PatternComponent-Level Splitting
Lazy load heavy components (modals, charts, rich editors) only when they're actually needed.
// Only loads chart lib when modal opens
const HeavyChartModal = lazy(() =>
  import('./HeavyChartModal'));

function Dashboard() {
  const [open, setOpen] = useState(false);
  return (
    <>
      <button onClick={() => setOpen(true)}>
        View Charts
      </button>
      {open &&
        <Suspense fallback={<Spinner />}>
          <HeavyChartModal />
        </Suspense>}
    </>
  );
}
PerfPreload on Hover
Start loading the next page's chunk when user hovers a link — by the time they click, it's ready.
const settingsLoader = () =>
  import('@/pages/SettingsPage');

<Link
  to="/settings"
  onMouseEnter={settingsLoader}
  onFocus={settingsLoader}
>
  Settings
</Link>

API Layer

Components should never call fetch() directly. All HTTP logic lives in a dedicated API layer that's completely separate from React. This makes it testable, swappable, and consistent.

PatternAxios Instance with Interceptors
// services/apiClient.ts
import axios from 'axios';

export const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  timeout: 10000,
  headers: { 'Content-Type': 'application/json' },
});

// Attach auth token on every request
apiClient.interceptors.request.use(config => {
  const token = getToken(); // from Zustand or localStorage
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Global error handling — auto-redirect on 401
apiClient.interceptors.response.use(
  res => res,
  async error => {
    if (error.response?.status === 401) {
      clearAuthAndRedirect();
    }
    return Promise.reject(error);
  }
);

// features/users/usersApi.ts — typed API functions
export const usersApi = {
  getAll: (): Promise<User[]> =>
    apiClient.get('/users').then(r => r.data),
  getById: (id: string): Promise<User> =>
    apiClient.get(`/users/${id}`).then(r => r.data),
  update: (id: string, data: Partial<User>) =>
    apiClient.patch(`/users/${id}`, data).then(r => r.data),
};
Rule3-Layer Architecture
Components call hooks → hooks call API functions → API functions call apiClient. Never skip a layer.
// Layer 1: Component
function UserList() {
  const { data } = useUsers(); // ← hook
}

// Layer 2: Hook
function useUsers() {
  return useQuery({
    queryFn: usersApi.getAll // ← api fn
  });
}

// Layer 3: API function
const usersApi = {
  getAll: () => apiClient.get('/users')
};

Error Handling

Production apps need 3 layers of error handling: Error Boundaries (React tree errors), React Query error states (API errors), and a global toast/notification system.

PatternError Boundary Per Route
Wrap each route in its own Error Boundary. One crash doesn't bring down the whole app.
function RouteErrorBoundary({ children }) {
  return (
    <ErrorBoundary
      fallback={<PageErrorFallback />}
      onError={(err, info) =>
        logToSentry(err, info)}
    >
      {children}
    </ErrorBoundary>
  );
}
PatternGlobal Error Toast (React Query)
Set a default error handler in QueryClient so every failed query shows a toast — without any per-hook try/catch.
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: 1,
      throwOnError: false,
    },
    mutations: {
      onError: (error) => {
        toast.error(
          error.message || 'Something went wrong'
        );
      },
    },
  },
});
PatternTyped API Errors
Create a typed error class for API errors so you can check error codes and display user-friendly messages.
class ApiError extends Error {
  constructor(
    public status: number,
    public code: string,
    message: string
  ) { super(message); }
}

function isApiError(e: unknown):
  e is ApiError {
  return e instanceof ApiError;
}

// Usage:
if (isApiError(error) &&
    error.status === 404) {
  return <NotFound />;
}

TypeScript Patterns

TypeScript is not optional at production scale. These patterns ensure type safety flows end-to-end without unnecessary any.

PatternDiscriminated Unions for State
Model async state as a discriminated union instead of separate booleans. Impossible states become impossible types.
// ❌ Boolean soup — invalid states possible
type Bad = {
  loading: boolean;
  error: Error | null;
  data: User[] | null;
}

// ✅ Discriminated union
type UserState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User[] }
  | { status: 'error'; error: Error };
PatternInfer Types from Zod Schemas
Define Zod schemas as the single source of truth. Infer TS types from them — never duplicate type definitions.
const userSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(['admin', 'user']),
});

// Type inferred — single source of truth
type User = z.infer<typeof userSchema>;
Don'tNever Use 'any'
Use unknown instead of any for values you don't know yet. unknown forces you to narrow the type before using it.
// ❌ any disables type checking
function handleError(e: any) {
  console.log(e.message); // no check
}

// ✅ unknown forces narrowing
function handleError(e: unknown) {
  if (e instanceof Error)
    console.log(e.message); // safe
}
PatternComponentPropsWithout for Extends
When building a wrapper component that extends a native element, use ComponentPropsWithoutRef to forward all native props.
interface ButtonProps
  extends ComponentPropsWithoutRef<'button'> {
  variant?: 'primary' | 'ghost';
  loading?: boolean;
}

export function Button({
  variant, loading, children,
  ...rest  // all native button props
}: ButtonProps) {
  return <button {...rest}>{children}</button>;
}

Testing Strategy

Test behavior, not implementation. Users click buttons, fill forms, and see text — your tests should do the same. Delete tests that mock internals or test implementation details.

Test TypeToolWhat to TestRatio
UnitVitestPure utils, formatters, validators, Zustand storesMany
IntegrationRTL + VitestComponents with interactions, forms, routingMost
E2EPlaywrightCritical user flows (login, checkout, core actions)Few
PatternrenderWithProviders Wrapper
Wrap RTL's render with your providers so every test works with full context. No more boilerplate per test.
// test/utils.tsx
function renderWithProviders(
  ui: ReactElement,
  options = {}
) {
  const queryClient = new QueryClient({
    defaultOptions: {
      queries: { retry: false }
    }
  });
  return render(
    <QueryClientProvider client={queryClient}>
      <MemoryRouter>
        {ui}
      </MemoryRouter>
    </QueryClientProvider>,
    options
  );
}
PatternMock API with MSW
Use Mock Service Worker (MSW) to intercept real HTTP requests in tests. Tests behave like real users with real API responses.
// test/handlers.ts
import { http, HttpResponse }
  from 'msw';

export const handlers = [
  http.get('/api/users', () =>
    HttpResponse.json([
      { id: '1', name: 'Alice' },
    ])
  ),
  http.post('/api/users', async ({ request }) => {
    const body = await request.json();
    return HttpResponse.json(
      { id: '2', ...body }, { status: 201 }
    );
  }),
];
✅
Testing rules of thumb: Query by role, label, or text — never by class name or test IDs unless absolutely necessary. Use userEvent not fireEvent for realistic interactions. Test what the user sees, not what the component stores internally.

Security & Auth

Never store JWTs in localStorage for sensitive apps — use httpOnly cookies. Implement auth at the router level, not per-component.

PatternAuth Context + Protected Routes
// features/auth/AuthProvider.tsx
const AuthContext = createContext<AuthCtx>(null!);

export function AuthProvider({ children }) {
  const { data: user, isLoading } =
    useQuery({
      queryKey: ['me'],
      queryFn: authApi.getMe,
      retry: false,
    });

  if (isLoading) return <FullPageSpinner />;

  return (
    <AuthContext.Provider value={{ user }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () =>
  useContext(AuthContext);

// ProtectedRoute.tsx
export function ProtectedRoute() {
  const { user } = useAuth();
  if (!user) return <Navigate to="/login"/>;
  return <Outlet />;
}
Don'tSecurity Checklist
  • Never use dangerouslySetInnerHTML with user input
  • Store sensitive tokens in httpOnly cookies, not localStorage
  • Validate ALL user input server-side (not just client Zod)
  • Use Content Security Policy (CSP) headers
  • Sanitize any user-generated HTML with DOMPurify
  • Verify token expiry on every protected route
  • Implement CSRF protection for mutation endpoints
PatternRole-Based Access
// hooks/usePermission.ts
export function usePermission(
  permission: Permission
) {
  const { user } = useAuth();
  return user?.permissions
    .includes(permission) ?? false;
}

// Component usage:
function AdminPanel() {
  const canEdit = usePermission('users:write');
  return (
    <div>
      {canEdit && <EditButton />}
    </div>
  );
}
