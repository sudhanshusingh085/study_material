The Decision Guide
State Management

useReducer

useState's more powerful sibling — for complex state with multiple actions

🧠 Real-World Analogy
Think of a bank account. You never set the balance directly — you send instructions: "deposit $100", "withdraw $50", "freeze account". The bank processes each instruction and updates the balance accordingly.
useReducer works exactly the same way. Instead of setting state directly, you dispatch actions (instructions). A reducer function (the bank) decides how the state changes based on each action. The state itself is never mutated directly.
📌 What is it?
A React hook that manages state through a dispatch → reducer → new state cycle. Instead of multiple setState calls, you send named action objects to one central function.
✅ Why use it?
When state has multiple sub-values that change together, or when next state depends on previous state in complex ways. Keeps all state logic in one predictable place.
⏰ When NOT to use?
Simple boolean toggles, single string/number state, or when useState is already clean and readable. Don't over-engineer. useState is fine for 80% of cases.
Step-by-Step Breakdown
1
Define your state shape and action types
First, think about what data you're managing and what things can happen to it. Each "thing that can happen" becomes an action type.
Define state + action types TypeScript
// The SHAPE of your state — what data do we have?
interface CartState {
  items: CartItem[];
  total: number;
  isLoading: boolean;
  error: string | null;
}

// The ACTIONS — things that can happen to this state
// Each action has a 'type' (what happened) and optional 'payload' (data)
type CartAction =
  | { type: 'ADD_ITEM';     payload: CartItem }
  | { type: 'REMOVE_ITEM';  payload: { id: string } }
  | { type: 'CLEAR_CART' }
  | { type: 'SET_LOADING';  payload: boolean }
  | { type: 'SET_ERROR';    payload: string };
2
Write the reducer function
The reducer is a pure function — it takes the current state + an action, and returns the new state. It never modifies state directly. Think of it as a switch that handles each action case.
The Reducer Function pure function — no side effects
// PURE means: same input → always same output, no API calls, no mutations
function cartReducer(
  state: CartState,
  action: CartAction
): CartState {

  switch (action.type) {

    case 'ADD_ITEM': {
      // ✅ Return a NEW object — never mutate `state` directly!
      const newItems = [...state.items, action.payload];
      return {
        ...state,                     // spread all existing state
        items: newItems,              // override only what changed
        total: calcTotal(newItems),
      };
    }

    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter(
          item => item.id !== action.payload.id
        ),
        total: calcTotal(state.items.filter(
          item => item.id !== action.payload.id
        )),
      };

    case 'CLEAR_CART':
      return { ...state, items: [], total: 0 };

    case 'SET_LOADING':
      return { ...state, isLoading: action.payload };

    default:
      return state; // always return state for unknown actions
  }
}
3
Use it in your component
useReducer returns [state, dispatch]. state is your current state object. dispatch is a function you call with an action object to trigger state changes.
Using in a Component ✅ Correct
const initialState: CartState = {
  items: [], total: 0, isLoading: false, error: null
};

function Cart() {
  // useReducer(reducerFunction, initialState)
  // Returns: [currentState, dispatchFunction]
  const [state, dispatch] = useReducer(cartReducer, initialState);

  // To change state, DISPATCH an action (never setState directly)
  const addItem = (item: CartItem) => {
    dispatch({ type: 'ADD_ITEM', payload: item });
    //         ↑ what happened    ↑ the data needed
  };

  const removeItem = (id: string) => {
    dispatch({ type: 'REMOVE_ITEM', payload: { id } });
  };

  const clearCart = () => {
    dispatch({ type: 'CLEAR_CART' });
    // No payload needed for CLEAR_CART
  };

  return (
    <div>
      <p>Items: {state.items.length} | Total: ${state.total}</p>
      <button onClick={clearCart}>Clear Cart</button>
    </div>
  );
}
❌ Don't — useState for complex state
// Scattered, hard to track what's changing with what
const [items, setItems] = useState([]);
const [total, setTotal] = useState(0);
const [loading, setLoading] = useState(false);

const addItem = (item) => {
  setItems(prev => [...prev, item]); // sync issue risk
  setTotal(prev => prev + item.price);
  // What if one fails? State is now inconsistent.
};
✅ Do — useReducer for related state
// All related state in one object, one action
const [state, dispatch] = useReducer(
  cartReducer, initialState
);

const addItem = (item) => {
  dispatch({ type: 'ADD_ITEM', payload: item });
  // Reducer handles items + total atomically
  // State is always consistent ✅
};
⚠️
The golden rule of reducers: Never mutate state directly. Always return a new object. state.items.push(item) is WRONG. return { ...state, items: [...state.items, item] } is correct. React detects changes by reference — mutating the same object won't trigger a re-render.
Zustand / Redux

useSelector

Subscribe to only the exact piece of global state your component needs

🧠 Real-World Analogy
Imagine a newspaper with 100 sections. Instead of re-reading the whole paper every time any section changes, you subscribe to ONLY the sports section. You only get notified when sports news changes — not when the weather or finance sections update.
useSelector is your subscription to a specific slice of global state. Without it (subscribing to the entire store), your component re-renders every time anything in the store changes — even data it doesn't use. With a selector, only relevant changes trigger re-renders.
📌 What is it?
A function you pass to Zustand's hook (or Redux's useSelector) that extracts a specific piece of state. The component only re-renders when that specific piece changes.
✅ Why it matters
Without selectors, every component subscribing to a store re-renders on every state change — even if the data they use didn't change. Selectors prevent wasted renders.
🚫 The wrong way
Destructuring the entire store: const { user, cart, ui } = useStore() subscribes to ALL state. Any change to ui re-renders a component that only uses user.
Zustand — Selector Pattern
The Store (Zustand)
// store/appStore.ts
interface AppStore {
  user: User | null;
  theme: 'light' | 'dark';
  sidebarOpen: boolean;
  notifications: Notification[];
  cartCount: number;
  // ...actions
  setUser: (user: User) => void;
  toggleSidebar: () => void;
}

export const useAppStore = create<AppStore>()((set) => ({
  user: null,
  theme: 'light',
  sidebarOpen: false,
  notifications: [],
  cartCount: 0,
  setUser: (user) => set({ user }),
  toggleSidebar: () => set(s => ({ sidebarOpen: !s.sidebarOpen })),
}));
❌ No Selector — Re-renders on anything
// NavBar only uses 'user' and 'theme'
// But this subscribes to the ENTIRE store
function NavBar() {
  // ⚠️ Re-renders when cartCount changes,
  // notifications change, sidebarOpen changes...
  // even though NavBar doesn't use any of those!
  const { user, theme } = useAppStore();

  return <nav>{user?.name}</nav>;
}
✅ With Selector — Only re-renders on changes it cares about
// NavBar ONLY re-renders when user or theme changes
// Cart updates, sidebar toggles = no re-render ✅
function NavBar() {
  // Pass a selector: s => the specific piece you need
  const user  = useAppStore(s => s.user);
  const theme = useAppStore(s => s.theme);

  return <nav>{user?.name}</nav>;
}
Derived / Computed Selectors selector can do calculations
// Selectors can compute derived values — not just pick raw state
function CartBadge() {
  // Compute something from state right inside the selector
  const itemCount = useCartStore(s => s.items.length);
  const hasItems  = useCartStore(s => s.items.length > 0);
  const totalOver100 = useCartStore(s => s.total > 100);

  return hasItems ? <span>{itemCount}</span> : null;
}

// For expensive computations, combine with useMemo
function ExpensiveComponent() {
  const items = useCartStore(s => s.items);
  const expensiveResult = useMemo(
    () => items.filter(i => i.active).sort(...),
    [items]
  );
}

React.memo

Prevents a component from re-rendering when its props haven't changed

🧠 Real-World Analogy
Think of a report printer. Every time the manager asks for a report, the printer checks — "did any of the data change since the last print?" If nothing changed, it hands back the cached copy without reprinting. Only if the data changed does it actually reprint.
React.memo wraps your component and remembers the last render result. If the parent re-renders but passes the same props, the component is skipped entirely. Only when props actually change does it re-render.
📌 What is it?
A Higher-Order Component (HOC) that wraps your component. React shallowly compares props before re-rendering. If all props are the same as last time, the render is skipped.
✅ When it helps
Component renders are expensive (heavy computation or large DOM) AND its parent re-renders frequently with data the child doesn't use. List items, table rows, repeated cards.
🚫 When it wastes time
Component is already fast, parent rarely re-renders, or props change every render anyway. memo has a comparison cost — it can be slower than just re-rendering for cheap components.
Without memo vs With memo — What happens on parent re-render
WITHOUT React.memo
Parent state changes
↓
Parent re-renders
↓
ALL child components re-render
↓
Even if their props didn't change ❌
WITH React.memo
Parent state changes
↓
Parent re-renders
↓
memo compares old props vs new props
↓
Same props → SKIP re-render ✅
Basic Usage ✅ Correct
// Wrap the component export with React.memo
const UserRow = React.memo(function UserRow({
  user,
  onDelete
}: {
  user: User;
  onDelete: (id: string) => void;
}) {
  console.log('UserRow rendered for:', user.name); // watch this in DevTools
  return (
    <tr>
      <td>{user.name}</td>
      <td>{user.email}</td>
      <button onClick={() => onDelete(user.id)}>Delete</button>
    </tr>
  );
}); // ← memo closes here

// Now in UserTable, when any unrelated state changes,
// UserRow only re-renders if its specific user or onDelete changed
function UserTable({ users, onDelete }) {
  return (
    <table>
      {users.map(user => (
        <UserRow key={user.id} user={user} onDelete={onDelete} />
      ))}
    </table>
  );
}
⚠️
The memo trap with functions as props: If you pass a function directly — onDelete={() => handleDelete(id)} — a new function is created on every render. This breaks memo because () => handleDelete(id) !== () => handleDelete(id) (different references). Fix: wrap the function with useCallback. That's why memo and useCallback always come together.
Performance

useMemo

Cache an expensive computed value — only recalculate when its inputs change

🧠 Real-World Analogy
You have a spreadsheet that takes 5 seconds to calculate a total across 10,000 rows. Instead of recalculating every time you look at it, you cache the result and only recalculate when the underlying data actually changes.
useMemo caches the return value of a function. On every render, React checks: "did the dependencies change?" If not, it skips running the function and returns the cached value from last time.
📌 What is it?
useMemo(fn, [deps]) — runs fn and caches its return value. Only re-runs when something in [deps] changes. Returns the cached result otherwise.
✅ Right use cases
Expensive calculations on large data (sorting/filtering 1000+ items). Creating stable object references to pass to memo'd children. Never use it just to "avoid creating an object".
🚫 Wrong use cases
Simple string concatenation, adding two numbers, accessing object properties. If the function runs in under 1ms, useMemo's overhead is bigger than the savings.
useMemo — Syntax and Real Examples
// SYNTAX: useMemo(function, dependencyArray)
//   - The function runs on first render
//   - It only re-runs when something in the [] changes
//   - The return value is what useMemo gives you back

function UserDashboard({ users, searchQuery, sortOrder }) {

  // ✅ GOOD — expensive filter + sort on large array
  // Only re-runs when users, searchQuery, or sortOrder changes
  const filteredAndSorted = useMemo(() => {
    console.log('Running expensive computation...'); // watch this
    return users
      .filter(u =>
        u.name.toLowerCase().includes(searchQuery.toLowerCase())
      )
      .sort((a, b) =>
        sortOrder === 'asc'
          ? a.name.localeCompare(b.name)
          : b.name.localeCompare(a.name)
      );
  }, [users, searchQuery, sortOrder]);
  //   ↑ Only re-compute when these 3 values change

  // ✅ GOOD — stable object reference for a memoized child
  const chartConfig = useMemo(() => ({
    type: 'bar',
    colors: ['#3b82f6', '#10b981'],
    gridLines: true,
  }), []); // empty [] = created once, never changes

  // ❌ BAD — this is NOT worth memoizing
  const title = useMemo(
    () => `Hello ${users.length} users`,
    [users.length]
  ); // just do: const title = `Hello ${users.length} users`

  return (
    <div>
      <ChartComponent config={chartConfig} />
      {filteredAndSorted.map(u => <UserRow key={u.id} user={u} />)}
    </div>
  );
}
Performance

useCallback

Cache a function reference — so it doesn't break memoized children on every render

🧠 Real-World Analogy
Every time you ask someone for their phone number, they hand you a new slip of paper with the same digits. Even though the number is identical, it's a different piece of paper. A computer checking "is this the same number?" sees two different papers and says "no, they're different."
In JavaScript, () => doSomething() creates a brand new function reference on every render — even if the code inside is identical. React.memo compares by reference, not by code content. useCallback gives you back the same reference between renders.
📌 What is it?
useCallback(fn, [deps]) — returns the same function reference between renders. Only creates a new function when deps change. Solves the "new function every render" problem.
✅ When you need it
When you pass a function as a prop to a React.memo'd component. When a function is in a useEffect dependency array. When a function is a dep for another useMemo/useCallback.
🚫 When you don't need it
Event handlers attached directly in JSX (not passed to children). Functions used only inside the same component. If the child isn't memo'd, useCallback does nothing useful.
useCallback — Full Explanation With Code
// The problem: without useCallback
function ParentList({ userIds }) {
  const [filter, setFilter] = useState('');

  // ❌ New function created on EVERY render of ParentList
  // Even when 'filter' state changes (which has nothing to do with delete)
  // This causes EVERY UserRow (memo'd) to re-render unnecessarily
  const handleDelete = (id: string) => {
    deleteUser(id);
  };

  // ✅ useCallback: returns same function unless dependencies change
  // Since there are no deps [], same reference forever
  // UserRow sees same onDelete prop → stays memoized ✅
  const handleDelete = useCallback((id: string) => {
    deleteUser(id);
  }, []); // empty [] = never recreate this function
  //     ↑ what the function depends on

  // Example with a dependency
  const handleUpdate = useCallback((id: string, data: object) => {
    updateUser(id, { ...data, updatedBy: currentUser.id });
    // 'currentUser' is used inside, so it must be in deps
  }, [currentUser]); // recreate only when currentUser changes

  return (
    <div>
      <input value={filter}
        onChange={e => setFilter(e.target.value)} />
      {userIds.map(id => (
        // UserRow is React.memo'd — it won't re-render
        // when filter changes, because handleDelete is stable ✅
        <UserRow key={id} userId={id}
          onDelete={handleDelete} />
      ))}
    </div>
  );
}
useMemo vs useCallback — The One Clear Difference
useMemo
Caches the return value of a function. Use when you want to cache a computed result (a number, array, object). useMemo(() => expensiveCalc(), [dep]) → gives you the result.
useCallback
Caches the function itself. Use when you want to keep the same function reference. useCallback(() => doSomething(), [dep]) → gives you the function.
Equivalent?
useCallback(fn, deps) is exactly the same as useMemo(() => fn, deps). useCallback is just a convenience shorthand for memoizing functions specifically.
React Query

staleTime

How long React Query considers data "fresh" before it will refetch in the background

🧠 Real-World Analogy
Think of a carton of milk with an expiry date. For staleTime minutes, the milk is "fresh" — you use it without questioning it. After that, it's "stale" — you'll check if there's fresher milk available, but you'll still drink the old one while waiting for delivery.
While data is "fresh" (within staleTime), React Query serves it from cache with no network request. Once it's "stale", React Query still shows the cached data immediately (no loading spinner!), but silently fetches fresh data in the background and updates the UI when it arrives.
📌 Default value
0 milliseconds — meaning data is considered stale immediately after fetching. Every time you focus a window or remount a component, React Query refetches. This is conservative but can be very noisy.
✅ What it controls
How often React Query refetches in the background. Higher = fewer network requests, potentially slightly stale data. Lower = more fresh data, more network requests. It's a freshness vs. cost tradeoff.
⏰ Key insight
staleTime does NOT remove data from memory. Stale data is still shown to the user while fresh data loads. It only controls when background refetching is triggered.
The Data Lifecycle Timeline
FRESH
Within staleTime (e.g., 0–5 min) — served from cache, NO network request at all
STALE
After staleTime — still shows cached data, but silently refetches in background on window focus / component mount
GONE
After gcTime (5 min default) — removed from memory. Next access shows loading state again
staleTime — Practical Settings for Different Data Types
// ────────────────────────────────────────────────────────────
// Rule of thumb: how fast does this data change in the real world?
// ────────────────────────────────────────────────────────────

// 1. REAL-TIME data (stock prices, live scores, chat)
// Changes every second → staleTime: 0 (default)
useQuery({
  queryKey: ['stock-price', ticker],
  queryFn: fetchPrice,
  staleTime: 0,       // Always refetch on focus/mount
  refetchInterval: 5000, // Also poll every 5 seconds
});

// 2. USER DATA (profile, settings, notifications)
// Changes occasionally → staleTime: 1–5 minutes
useQuery({
  queryKey: ['user-profile', userId],
  queryFn: fetchUserProfile,
  staleTime: 5 * 60 * 1000, // 5 minutes = 300,000 ms
});

// 3. REFERENCE DATA (countries, categories, config)
// Changes rarely or never → staleTime: Infinity
useQuery({
  queryKey: ['countries'],
  queryFn: fetchCountries,
  staleTime: Infinity, // Fetch once, never refetch
});

// 4. Set a default for the whole app in QueryClient
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000, // 1 min default for all queries
    },
  },
});
React Query

gcTime (formerly cacheTime)

How long unused data stays in memory before being garbage collected

🧠 Real-World Analogy
It's like your browser's back/forward cache. When you navigate away from a page, the page data stays in memory for a while — so if you come back quickly, it loads instantly. But if you stay away long enough, it's cleared to free up memory.
gcTime controls how long data stays in the React Query cache after no component is using it anymore. If you navigate back before gcTime expires, you get instant data + background refresh. After gcTime, the cache is cleared and the next visit shows a loading spinner.
staleTime vs gcTime — Cleared Up Once and For All
staleTime
Controls when to refetch in the background. Data is still shown to user. It's a freshness question: "should I fetch newer data while showing this cached version?"
gcTime
Controls when to DELETE data from memory. Once no component uses the data and gcTime expires, it's gone. Next visit = loading spinner. It's a memory management question.
Rule
gcTime must always be ≥ staleTime. There's no point keeping stale data in cache for less time than it's considered fresh. Default: staleTime=0, gcTime=5min.
Old name
cacheTime was renamed to gcTime in React Query v5 (TanStack Query) to be clearer about its purpose. If you see cacheTime in old docs, it means gcTime.
gcTime — Real Example
// Scenario: User navigates Dashboard → Settings → back to Dashboard
// With gcTime: 5min (default), Dashboard data is still in cache
// → instant load with background refresh ✅

useQuery({
  queryKey: ['dashboard'],
  queryFn: fetchDashboard,
  staleTime: 60 * 1000,  // 1 min — show cache, don't refetch within 1 min
  gcTime: 10 * 60 * 1000, // 10 min — keep in memory for 10 min after unmount
});

// For very large datasets you don't want in memory:
useQuery({
  queryKey: ['huge-report'],
  queryFn: fetchReport,
  gcTime: 30 * 1000,  // Only keep for 30 seconds after unmount
});

// For always-needed reference data:
useQuery({
  queryKey: ['app-config'],
  queryFn: fetchConfig,
  staleTime: Infinity,
  gcTime: Infinity,   // Never garbage collect — keep forever
});
Custom Hook

useDebounce

Delay updating a value until the user stops changing it — prevents spamming APIs

🧠 Real-World Analogy
Imagine an elevator with a "close door" button. Pressing it once doesn't close the door — the elevator waits to see if anyone else is coming. Only after no one presses anything for 3 seconds does it close. Each button press resets the 3-second timer.
Debouncing works the same way. Every time the value changes (user types), the timer resets. Only after the user stops changing the value for the specified delay (e.g., 400ms) does the debounced value actually update — and the search/API call fires.
📌 The Problem It Solves
User types "react hooks" — that's 11 keystrokes. Without debounce: 11 API calls. With debounce(400ms): 1 API call after they stop typing. Saves money, reduces server load, prevents race conditions.
✅ When to use
Search inputs, autocomplete, real-time form validation, resize handlers, any input that triggers expensive operations on every keystroke.
⏰ Delay guide
150–200ms for fast local filtering. 300–400ms for API searches (most common). 500–700ms for expensive server operations. Too short = still spammy. Too long = feels laggy.
Complete useDebounce — Line by Line
1
The hook itself — what each line does
useDebounce.ts — Fully Annotated
function useDebounce<T>(
  value: T,      // The value to debounce (can be any type: string, number, object)
  delay: number  // How long to wait after last change (in milliseconds)
): T {
  // debouncedValue starts as a copy of the current value
  const [debouncedValue, setDebouncedValue] = useState<T>(value);

  useEffect(() => {
    // Step 1: Set a timer. After 'delay' ms, update debouncedValue
    const timer = setTimeout(() => {
      setDebouncedValue(value); // ← This triggers re-render with new value
    }, delay);

    // Step 2: CLEANUP — this is the key part!
    // If 'value' changes BEFORE the timer fires (user types another char),
    // this cleanup runs FIRST, canceling the previous timer.
    // Then a new timer starts from Step 1.
    // Result: only the last timer (after user stops typing) actually fires.
    return () => clearTimeout(timer);

  }, [value, delay]);
  //   ↑ Re-run effect whenever value or delay changes

  return debouncedValue;
}
2
Using it in a search component
Search Component — With and Without Debounce
function SearchUsers() {
  const [query, setQuery] = useState('');

  // 'query' = what's in the input right now (updates every keystroke)
  // 'debouncedQuery' = only updates after user pauses 400ms
  const debouncedQuery = useDebounce(query, 400);

  // This API call only fires when debouncedQuery changes
  // NOT on every keystroke. ✅
  const { data: users, isLoading } = useQuery({
    queryKey: ['users', debouncedQuery], // key includes debounced value
    queryFn: () => searchUsers(debouncedQuery),
    enabled: debouncedQuery.length > 2, // only search if at least 3 chars
  });

  return (
    <div>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="Search users..."
      />
      {/* Show spinner while typing (query !== debouncedQuery) */}
      {query !== debouncedQuery && <Spinner />}
      <UserList users={users ?? []} loading={isLoading} />
    </div>
  );
}
Custom Hook

useClickOutside

Run a callback when the user clicks anywhere outside a specific element

🧠 Real-World Analogy
Like a lightbox gallery. When you click outside the image, it closes. The gallery "knows" whether you clicked inside the photo or on the dark background around it.
useClickOutside attaches a global click listener that checks every click: "did it happen inside my element, or somewhere else?" If outside — call the handler (typically close something).
📌 What it solves
Closing dropdowns, modals, popovers, context menus, date pickers when the user clicks outside them. Without this hook, menus stay open forever unless you click the exact close button.
✅ Why not just blur?
onBlur only fires for focusable elements. Non-focusable elements like divs don't receive blur. Also, blur fires before click — causing the click to be lost. Click-outside is more reliable.
⏰ Common uses
Dropdown menus, custom selects, date pickers, color pickers, context menus, tooltip panels, search suggestion lists.
Complete useClickOutside — Line by Line
1
The hook — fully annotated
useClickOutside.ts
import { useEffect, useRef } from 'react';

function useClickOutside<T extends HTMLElement>(
  handler: () => void  // function to call when clicking outside
) {
  // useRef gives us a stable reference to the DOM element
  // We'll attach this ref to the element we want to "watch"
  const ref = useRef<T>(null);

  useEffect(() => {
    const handleClick = (event: MouseEvent) => {
      // ref.current = the actual DOM element
      // .contains(event.target) = was the click INSIDE our element?
      if (
        ref.current &&
        !ref.current.contains(event.target as Node)
        // If ref exists AND the click was NOT inside ref
      ) {
        handler(); // ← click was outside → call the handler
      }
      // If click was inside ref.current → do nothing
    };

    // Listen on the whole document for any mousedown event
    // mousedown fires before click — better for closing patterns
    document.addEventListener('mousedown', handleClick);

    // Cleanup: remove listener when component unmounts
    // (or when handler changes and effect re-runs)
    return () => {
      document.removeEventListener('mousedown', handleClick);
    };
  }, [handler]); // Re-run if handler function reference changes

  return ref; // Return the ref so the component can attach it
}
2
Using it in a dropdown component
Dropdown with useClickOutside
function Dropdown({ options, onSelect }) {
  const [isOpen, setIsOpen] = useState(false);

  // Wrap handler in useCallback so its reference stays stable
  // (prevents the useEffect inside useClickOutside from re-running)
  const close = useCallback(() => setIsOpen(false), []);

  // ref is what we attach to the dropdown element
  // Clicking anywhere outside the element with this ref → close() fires
  const dropdownRef = useClickOutside<HTMLDivElement>(close);

  return (
    {/* Attach ref to the container we want to "watch" */}
    <div ref={dropdownRef} className="relative">
      <button onClick={() => setIsOpen(o => !o)}>
        Open Menu
      </button>

      {isOpen && (
        <ul className="absolute top-full mt-1 bg-white shadow-lg">
          {options.map(opt => (
            <li key={opt.value}
              onClick={() => { onSelect(opt); setIsOpen(false); }}>
              {opt.label}
            </li>
          ))}
        </ul>
      )}
      {/* Click inside div → nothing. Click outside → close() */}
    </div>
  );
}

usePrevious

Remember what a value was on the previous render

🧠 Real-World Analogy
Like writing down today's temperature before checking tomorrow's forecast — you always know what it was yesterday to compare with today.
📌 What it does
Returns the value of a variable from the previous render. On first render it's undefined. From render 2 onward, it holds what the value was in render 1, 2, 3...
✅ Use cases
Animate when a value increases vs decreases (score going up = green flash, down = red flash). Log changes. Detect direction of change. Compare before/after in forms.
💡 How it works
Uses a ref to store the previous value. A ref persists between renders but changing it does NOT trigger a re-render — making it perfect for "memory" without causing extra renders.
usePrevious — Full Implementation + Usage
// The hook: just 7 lines
function usePrevious<T>(value: T): T | undefined {
  // useRef persists a value between renders WITHOUT causing re-renders
  const ref = useRef<T>();

  useEffect(() => {
    // After every render: save current value into the ref
    // This runs AFTER the component renders with the new value
    // So during render, ref.current still = the PREVIOUS value
    ref.current = value;
  }); // No deps [] = runs after every render

  return ref.current; // Returns what value was on the LAST render
}

// ─────── Usage Example ───────
function ScoreDisplay({ score }: { score: number }) {
  const prevScore = usePrevious(score);

  // Determine if score went up, down, or stayed same
  const direction = prevScore === undefined
    ? 'neutral'
    : score > prevScore ? 'up'
    : score < prevScore ? 'down'
    : 'neutral';

  return (
    <div className={
      direction === 'up'   ? 'text-green-600' :
      direction === 'down' ? 'text-red-600'   :
      'text-gray-900'
    }>
      {score}
      {prevScore !== undefined && (
        <span>
          {direction === 'up' ? '↑' : direction === 'down' ? '↓' : ''}
          {Math.abs(score - prevScore)}
        </span>
      )}
    </div>
  );
}

useAsync

Track the loading/error/data state of any one-off async operation

🧠 Real-World Analogy
Like a delivery tracking dashboard. You can see: "Package is being prepared" (idle), "In transit" (loading), "Delivered!" (success with data), or "Delivery failed" (error). All from one place.
📌 useAsync vs React Query
React Query is for data fetching that should be cached (GET requests, often re-fetched). useAsync is for one-off actions that don't need caching (button-triggered POST, file upload, export generation).
✅ When to use useAsync
Submit a form, export a CSV, upload a file, send an email, trigger a payment. Actions the user initiates manually. Results you don't need to cache or share across components.
🚫 When NOT to use
Fetching data that's shown in the UI on load. Use React Query's useQuery for that — you get caching, background refetch, and deduplication for free.
useAsync — Full Implementation + Usage
// useAsync.ts
type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error };

function useAsync<T>() {
  const [state, setState] = useState<AsyncState<T>>(
    { status: 'idle' }
  );

  // 'execute' is what you call to trigger the async operation
  // Pass any async function — it handles loading/error/success
  const execute = useCallback(
    async (asyncFn: () => Promise<T>) => {
      setState({ status: 'loading' });
      try {
        const data = await asyncFn();
        setState({ status: 'success', data });
        return data;
      } catch (error) {
        setState({
          status: 'error',
          error: error instanceof Error
            ? error : new Error('Unknown error')
        });
        throw error;
      }
    }, []
  );

  const reset = useCallback(
    () => setState({ status: 'idle' }), []
  );

  return { ...state, execute, reset };
}

// ─────── Usage in a Component ───────
function ExportButton({ reportId }: { reportId: string }) {
  const { status, execute } = useAsync<Blob>();

  const handleExport = () => {
    execute(() => exportReport(reportId))
      .then(blob => downloadFile(blob, 'report.pdf'))
      .catch(() => {}); // errors already handled in state
  };

  return (
    <button
      onClick={handleExport}
      disabled={status === 'loading'}
    >
      {status === 'loading' ? 'Exporting...'
       : status === 'error'   ? 'Export Failed — Retry'
       : status === 'success' ? 'Downloaded! ✓'
       : 'Export PDF'}
    </button>
  );
}

useImperativeHandle

Expose specific methods from a child component to a parent using a ref

🧠 Real-World Analogy
A car has hundreds of internal systems, but the dashboard exposes only 3 controls to the driver: ignition, accelerator, brakes. You don't expose the engine internals — just the interface the driver needs. useImperativeHandle is your component's dashboard.
Normally, a ref on a component gives the parent access to the raw DOM node. useImperativeHandle lets you REPLACE that with a custom object of only the methods you want to expose — keeping implementation details private.
📌 What it does
When a parent uses a ref on a child component, it normally gets the DOM element. useImperativeHandle lets the child decide what the parent sees through that ref — exposing only specific methods.
✅ Real use cases
A <Modal /> that the parent can ref.open() and ref.close(). An <AudioPlayer /> the parent can ref.play() and ref.pause(). A <TextEditor /> with ref.focus() and ref.getContent().
🚫 When to avoid
Most of the time! If you can solve it with props and callbacks, do that. useImperativeHandle is an escape hatch for cases where the parent needs to imperatively trigger something in the child.
useImperativeHandle — Modal Example
// Step 1: Define what methods you want to expose
interface ModalHandle {
  open:  () => void;
  close: () => void;
}

// Step 2: Wrap the child with forwardRef to receive the ref
const Modal = forwardRef<ModalHandle, { title: string; children: ReactNode }>(
  ({ title, children }, ref) => {
    const [isOpen, setIsOpen] = useState(false);

    // Step 3: useImperativeHandle defines what 'ref' shows to the parent
    useImperativeHandle(ref, () => ({
      open:  () => setIsOpen(true),
      close: () => setIsOpen(false),
      // Parent does NOT have access to isOpen, setIsOpen, or anything else
      // Only what you put inside this object ↑
    }));

    if (!isOpen) return null;
    return (
      <div className="fixed inset-0 z-50 flex items-center justify-center">
        <div className="bg-white rounded-2xl p-6 shadow-2xl">
          <h2>{title}</h2>
          {children}
        </div>
      </div>
    );
  }
);

// Step 4: Parent uses a ref to control the Modal imperatively
function Dashboard() {
  const confirmRef = useRef<ModalHandle>(null);

  return (
    <div>
      <button onClick={() => confirmRef.current?.open()}>
        Delete Account
      </button>

      <Modal ref={confirmRef} title="Confirm Deletion">
        <p>Are you sure?</p>
        <button onClick={() => confirmRef.current?.close()}>Cancel</button>
      </Modal>
    </div>
  );
}

useContext — Deep Dive

Share data through the component tree without prop drilling — but use it carefully

🧠 Real-World Analogy
Like a public announcement system in a building. Instead of one person running to every floor to tell each employee "today is casual Friday" (prop drilling), the announcement plays once and every floor (every component) hears it simultaneously.
📌 What it does
A Provider wraps your component tree and provides a value. Any component below it in the tree can read that value with useContext(MyContext) — no props needed at all.
✅ Good for
Rarely-changing global data: current logged-in user, theme (light/dark), locale/language, feature flags. Data that many components need but is stable.
🚫 Bad for
Frequently-changing data. Every component using the context re-renders when the value changes — even if they only use a small part of it. Use Zustand for this instead.
Production Context Pattern — With Custom Hook
// ─── 1. Create the context ───
// Pass null as default, but use a type union for safety
interface AuthContextType {
  user: User | null;
  login:  (credentials: Credentials) => Promise<void>;
  logout: () => void;
}

const AuthContext = createContext<AuthContextType | null>(null);

// ─── 2. Create the Provider component ───
export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);

  const login = useCallback(async (credentials) => {
    const user = await authApi.login(credentials);
    setUser(user);
  }, []);

  const logout = useCallback(() => {
    setUser(null);
    authApi.logout();
  }, []);

  // useMemo prevents new object on every render (important for performance)
  const value = useMemo(
    () => ({ user, login, logout }),
    [user, login, logout]
  );

  return (
    <AuthContext.Provider value={value}>
      {children}
    </AuthContext.Provider>
  );
}

// ─── 3. Create a custom hook (ALWAYS do this — never use useContext directly) ───
// This validates the context is used inside the Provider
// AND gives a clear error message if not
export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error('useAuth must be used within AuthProvider');
  }
  return context;
}

// ─── 4. Usage in any component anywhere in the tree ───
function UserMenu() {
  const { user, logout } = useAuth(); // No prop drilling! ✅
  return (
    <div>
      <span>{user?.name}</span>
      <button onClick={logout}>Sign Out</button>
    </div>
  );
}

useRef — The 3 Use Cases

A ref is a box that holds a value — changing it never causes a re-render

🧠 Real-World Analogy
Like a sticky note on your monitor. You can write on it, change it, read it — but updating the sticky note doesn't trigger any alerts or notifications. It's just quietly available whenever you need to check it.
useState triggers a re-render when changed. useRef does NOT. That's the entire difference. A ref is just a mutable container that persists between renders silently.
useRef — All 3 Use Cases Explained
// ═══════════════════════════════════════════════════
// USE CASE 1: Access a DOM element directly
// ═══════════════════════════════════════════════════
function AutoFocusInput() {
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    // After render, the DOM element is available via .current
    inputRef.current?.focus(); // Directly call DOM methods
  }, []);

  return <input ref={inputRef} placeholder="Auto-focused!" />;
}

// ═══════════════════════════════════════════════════
// USE CASE 2: Store a mutable value that persists between renders
// WITHOUT causing re-renders when it changes
// ═══════════════════════════════════════════════════
function Timer() {
  const [time, setTime] = useState(0);

  // Store the interval ID in a ref (not state)
  // Because: we don't want to re-render when intervalId changes
  // We just need to remember it so we can clear it later
  const intervalRef = useRef<NodeJS.Timeout>();

  const start = () => {
    intervalRef.current = setInterval(
      () => setTime(t => t + 1), 1000
    );
  };

  const stop = () => {
    clearInterval(intervalRef.current);
  };

  return (
    <div>
      <p>{time}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}

// ═══════════════════════════════════════════════════
// USE CASE 3: Keep the latest value of something in a callback
// (avoids stale closure issues)
// ═══════════════════════════════════════════════════
function useEventCallback<T extends (...args: any[]) => any>(fn: T): T {
  const ref = useRef(fn);

  // Always keep ref.current pointing to the latest version of fn
  useEffect(() => { ref.current = fn; });

  // Return a stable function that calls the latest fn
  return useCallback(
    (...args) => ref.current(...args), []
  ) as T;
}
useState vs useRef — When to Choose Which
Use useState when
The value is shown in the UI. Changing it should update what the user sees. Examples: input value, toggle state, count, list of items.
Use useRef when
The value is not shown in the UI and you don't want a re-render when it changes. Examples: timer IDs, previous values, DOM element references, scroll position.

The Decision Guide

When to use each hook — the complete cheat sheet

Hook / ToolUse WhenDon't Use When
useState Simple, independent values (boolean, string, number, small objects). The bread and butter of React state. Multiple related values that change together, or state with complex transitions.
useReducer Complex state with multiple sub-values, state transitions have names (ADD_ITEM, REMOVE_ITEM), or next state depends on multiple parts of current state. Simple boolean toggles or single values. Don't over-engineer.
useContext Global data that many components need but rarely changes: auth user, theme, locale, feature flags. Frequently changing data — use Zustand instead. Context re-renders all consumers on any change.
Zustand store Global UI state that changes often: sidebar open/closed, notifications, cart count, theme toggling. Server data — use React Query. Per-component state — use useState.
React Query Any data that comes from a server. Literally everything that involves fetch/axios. It gives you caching, loading states, and background refetch for free. Client-only UI state that never touches an API.
React.memo A component is expensive to render AND its parent re-renders often. Use React DevTools Profiler to confirm it's actually slow first. Cheap components, rarely re-rendering parents. Memo has a comparison cost — it can be slower for trivial components.
useMemo Expensive computation (filtering/sorting large arrays). Creating stable object/array references for memo'd children. Simple expressions. String concatenation. Accessing object properties. The overhead isn't worth it.
useCallback A function is passed as a prop to a React.memo'd child. A function is a useEffect dependency. Event handlers used only in the same component. Functions not passed to children. Useless without memo'd children.
useRef Access DOM elements. Store mutable values that DON'T need to trigger re-renders (timer IDs, counters, scroll positions). Values that should update the UI when changed — use useState for those.
useDebounce Any input that triggers a network request or expensive computation on each keystroke. Search, autocomplete, real-time validation. Instant local filtering of small arrays. No need to debounce if it's just filtering 10 items in memory.
useClickOutside Any element that should close when user clicks away: dropdowns, popovers, context menus, date pickers, search suggestions. Modals with a visible overlay (use the overlay's onClick instead). Full-screen elements with no "outside".
🎯
The simplest rule: Always start with the simplest tool. useState first. Only upgrade to useReducer, Context, or Zustand when useState becomes painful. Only add memo/useMemo/useCallback when you have measured a real performance problem. Complexity added before it's needed is always a net negative.
