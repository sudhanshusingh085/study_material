From Scratch

You've built real React apps. You know useState and props. Now I'm going to teach you Context and Redux the way a senior dev would explain it — starting with the problem, then building each piece one at a time until you can write it yourself.

What you'll be able to do after this: Build a Context from scratch without googling. Set up a Redux store from scratch without AI help. Know which one to reach for and why. Understand every single line you write.

01The Problem

Why does any of this exist?

Before learning Context or Redux, you need to understand the exact pain they solve. Because if you don't feel that pain, the solution won't make sense.

You already know how React data flows: parent passes data down to child via props. That's it. That's the only built-in mechanism.

📦
Think of it like a relay race

Each runner (component) must receive the baton (data) from the previous runner and hand it to the next. If the data needs to reach a runner at the end of the chain, every runner in between must carry it — even if they don't need it personally.

This works great for small component trees. The problem starts when you have data that's needed by many different components scattered across your app — like the logged-in user's info. Where does that data live?

02The Problem

Prop Drilling — the real pain

Imagine your app needs to show the user's name in the Header, their avatar in the Sidebar, and their ID to fetch orders in the OrdersPage. The natural solution: put user state at the top and pass it down.

├─

Here's what the code looks like. Read this carefully — notice how Layout, Navbar, and Content are forced to handle a prop they never use:

// App.jsx — user state lives here
function App() {
  const [user, setUser] = useState({ name: 'Alice', avatar: '...', id: 'u1' });
  return <Layout user={user} />;  // must pass user down
}

// Layout.jsx — Layout has NOTHING to do with user
// But it must accept the prop just to pass it further
function Layout({ user }) {           // ← forced to accept user
  return (
    <div>
      <Navbar user={user} />         // ← forced to pass it on
      <Content user={user} />        // ← forced to pass it on
    </div>
  );
}

// Navbar.jsx — same problem
function Navbar({ user }) {           // ← forced to accept user
  return <Header user={user} />;    // ← forced to pass it on
}

// Header — finally uses it. 3 hops later.
function Header({ user }) {
  return <h1>Hello {user.name}</h1>;  // ← FINALLY used
}
prop-drilling.jsx
Why this is genuinely painful:

1. Every middle component is cluttered with props it doesn't care about. 2. If you rename the prop, you have to update every file in the chain. 3. Add a new component anywhere in the tree? You must thread the prop through again. 4. At 5+ levels deep, this becomes unmanageable.

The solution both Context and Redux provide:

A way for any component to reach data directly — without it being passed through every parent. Header can ask "give me the user" and get it, without Layout or Navbar ever knowing it exists.

03React Context

What Context actually is

📻
Context is a radio broadcast

You set up a radio station (the Provider) that broadcasts on a specific frequency (the Context object). Any component anywhere in the tree can tune in (useContext) and receive whatever the station is currently broadcasting. When the broadcast changes, every tuned-in component updates.

Context is built into React — no installation needed. It lets you make a value available to any component in your tree, without passing it through props at every level.

Here's the high-level picture before we go piece by piece:

How Context replaces prop drilling
UserContext.Provider — broadcasts user
├─
Layout — doesn't know about user at all
├─
Navbar — doesn't know about user at all
└─
Header — tunes in, gets user directly ✓
└─
Content — doesn't know about user at all
├─
Sidebar — tunes in, gets user directly ✓
└─
OrdersPage — tunes in, gets user directly ✓

Layout, Navbar, and Content now have zero knowledge of user. The components that need it get it directly from the Provider above them.

04React Context

The 3 pieces you always write

Every time you use Context, you write exactly 3 things. Not 2, not 4. Always 3. Once you know what each one does, the pattern becomes automatic.

The 3 pieces

① createContext()
Creates the "frequency" — the context object that ties Provider and consumers together
② Provider component
Wraps your app/subtree. Holds the state. Broadcasts the value down the tree.
③ useContext hook
Used inside any consumer component to tune in and get the value

Where each one lives

context/UserContext.jsx — ① and ② live here together. This is the file you create.
main.jsx — you wrap your App with ② the Provider here
Any component file — you use ③ here to read the value

Now let's understand each one deeply — not just what it does, but why it needs to exist.

05React Context

① createContext — what is it?

Why do we need to call createContext() at all? Why not just useContext directly?
Because you could have many different contexts in one app — UserContext, ThemeContext, CartContext. createContext() creates a unique "channel" object. Without it, React wouldn't know which channel you're trying to tune into.

createContext() returns an object. That object has two things on it: a .Provider and a .Consumer (we don't use Consumer — we use the hook instead). You call it once, at the top of your context file, and export the result.

// This is ALL createContext does — creates a unique channel object
const UserContext = createContext(null);

// The argument (null here) is the DEFAULT VALUE —
// only used if a component tries to read this context
// WITHOUT a Provider above it in the tree.
// Usually null or undefined. We'll handle the "no provider" case
// ourselves with an error message.

// UserContext now looks like this internally:
// {
//   Provider: [special React component],
//   Consumer: [special React component],
//   _currentValue: null  ← starts as your default
// }

createContext.js
Key insight:

The context object you get from createContext() is just a "key". It's the shared key that connects your Provider (which sets the value) and your consumers (which read the value). Think of it like a TV channel number — Provider broadcasts on channel 5, consumers tune to channel 5.

06React Context

② The Provider — the most important piece

This is the piece that was missing from your understanding. Let's go deep.

What is a Provider? Is it special React magic?
No. A Provider is just a React component — one that you wrap around other components. It receives a "value" prop and makes that value available to everything inside it. That's literally all it does.

The Provider has two responsibilities:

  • Hold the state — it uses useState internally to store whatever data you're sharing
  • Broadcast it — it wraps children in YourContext.Provider value={...} which makes the data available to any consumer below
This is the part that confuses people most:

You're creating TWO things with the same name pattern. UserContext is the object from createContext(). UserContext.Provider is a built-in React component that lives on that object. You're writing a wrapper component (let's call it UserProvider) that uses UserContext.Provider inside it. Your wrapper holds state; UserContext.Provider broadcasts it.

// Let's build the Provider piece by piece

// STEP 1: Your wrapper component
export function UserProvider({ children }) {
  // ↑ children = everything you wrap this around
  // We use { children } because this component will wrap other components

  // STEP 2: Put your state here — this is the state you want to share
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(false);

  // STEP 3: The built-in UserContext.Provider broadcasts your value
  // value={...} is what every consumer will receive when they call useContext
  return (
    <UserContext.Provider value={{ user, setUser, loading, setLoading }}>
      {children}
      {/* ↑ everything you wrap this around renders here */}
    </UserContext.Provider>
  );
}

UserProvider.jsx
What is "children" — I keep seeing it but I don't fully get it
When you write <UserProvider><App /></UserProvider>, the <App /> between the tags is automatically passed in as a prop called "children". Every React component can receive children. It just means "whatever is nested inside me". By rendering {children} in your Provider, you're saying "render whatever was nested inside me here".
// main.jsx — this is how you USE the Provider
import { UserProvider } from './context/UserContext';

createRoot(document.getElementById('root')).render(
  <UserProvider>       // ← wraps everything
    <App />              // ← App is "children" — gets rendered inside Provider
  </UserProvider>
);

// What this means: every component inside <App> and all its descendants
// can now access the user state — without any props being passed.

main.jsx
Why wrap at the top level?

You only need to wrap as high as the highest component that needs the data. If only components inside the Dashboard need user data, wrap just Dashboard. If the whole app needs it (which is common for auth), wrap at the root. The Provider creates a "bubble" — everything inside can access the value, everything outside cannot.

Adding functions to the Provider

Your Provider doesn't just share data — it should also share the functions to change that data. This keeps your logic in one place.

export function UserProvider({ children }) {
  const [user, setUser]       = useState(null);
  const [loading, setLoading] = useState(false);

  // Define functions here — they close over the state above
  const login = async (email, password) => {
    setLoading(true);
    try {
      const response = await fetch('/api/login', {
        method: 'POST',
        body: JSON.stringify({ email, password }),
        headers: { 'Content-Type': 'application/json' }
      });
      const data = await response.json();
      setUser(data.user);      // updates state → rerenders consumers
    } finally {
      setLoading(false);
    }
  };

  const logout = () => setUser(null);

  // Everything you include in "value" is available to consumers
  return (
    <UserContext.Provider value={{ user, loading, login, logout }}>
      {children}
    </UserContext.Provider>
  );
}

07React Context

③ useContext — how consumers read

This is the easy part. Any component that wants to read from the context just calls useContext(UserContext). React looks up the tree, finds the nearest UserContext.Provider, and returns whatever its value prop is.

// Any component, anywhere in the tree
import { useContext } from 'react';
import { UserContext } from './context/UserContext'; // import the context OBJECT

function Header() {
  // useContext returns the entire "value" you passed to the Provider
  const { user, logout } = useContext(UserContext);
  // ↑ destructure just the pieces you need from the value object

  return (
    <header>
      <span>Hello, {user?.name}</span>
      <button onClick={logout}>Log out</button>
    </header>
  );
}
// No props. No drilling. Header just asks for what it needs directly.

Header.jsx

The custom hook pattern — always do this

Instead of calling useContext(UserContext) directly in every component, you create a custom hook. This is not optional style — it's the correct pattern for two reasons: you get a better error message if the Provider is missing, and consumers import one clean name instead of two things.

// In your context file — add this function
export function useUser() {
  const context = useContext(UserContext);

  // This check saves you hours of debugging.
  // If you forgot to wrap with <UserProvider>, you get a clear error message
  // instead of a cryptic "Cannot read property 'name' of undefined"
  if (context === null) {
    throw new Error('useUser must be used inside a <UserProvider>. Did you forget to wrap?');
  }

  return context;
}

// Now components import and use ONE thing — much cleaner
import { useUser } from './context/UserContext';

function Header() {
  const { user, logout } = useUser(); // ← clean, obvious, safe
  return <header>Hello {user?.name}</header>;
}

08React Context

All 3 pieces together

Now let's see the complete Context setup in the exact order you build it. Three files, that's all.

// ─────────────────────────────────────────
// FILE 1: src/context/UserContext.jsx
// ─────────────────────────────────────────
import { createContext, useContext, useState } from 'react';

// ① Create the context object (the "channel")
const UserContext = createContext(null);

// ② The Provider — holds state, broadcasts it
export function UserProvider({ children }) {
  const [user, setUser]       = useState(null);
  const [loading, setLoading] = useState(false);

  const login = async (email, password) => {
    setLoading(true);
    const res  = await fetch('/api/login', { method: 'POST', body: JSON.stringify({ email, password }) });
    const data = await res.json();
    setUser(data.user);
    setLoading(false);
  };

  const logout = () => setUser(null);

  return (
    <UserContext.Provider value={{ user, loading, login, logout }}>
      {children}
    </UserContext.Provider>
  );
}

// ③ Custom hook — the clean way for components to read this context
export function useUser() {
  const ctx = useContext(UserContext);
  if (!ctx) throw new Error('useUser must be inside <UserProvider>');
  return ctx;
}

Context copy-paste template

Every time you create a new Context, copy this. Replace the word "User" with whatever you're sharing (Theme, Cart, Auth, etc.). Replace the state and functions with yours.

import { createContext, useContext, useState } from 'react';

// 1. Create the context
const XxxContext = createContext(null);

// 2. Build the Provider
export function XxxProvider({ children }) {
  // — your state here —
  const [yourData, setYourData] = useState(null);

  // — your functions here —
  const doSomething = () => { /* ... */ };

  // — everything consumers can access —
  return (
    <XxxContext.Provider value={{ yourData, doSomething }}>
      {children}
    </XxxContext.Provider>
  );
}

// 3. Custom hook for consuming
export function useXxx() {
  const ctx = useContext(XxxContext);
  if (!ctx) throw new Error('useXxx must be inside <XxxProvider>');
  return ctx;
}

✓ Context checklist — before you consider it done

  • Created the context object with createContext(null)
  • Provider component accepts {children} as prop
  • State lives inside the Provider with useState
  • Provider returns <XxxContext.Provider value={...}>{children}</XxxContext.Provider>
  • Custom hook with null-check wraps useContext
  • Provider wraps your App (or relevant subtree) in main.jsx
  • Consumers import and use the custom hook — not useContext directly
10React Context

The performance trap you need to know

Context has one significant limitation you must understand, or you'll create subtle performance bugs.

The rule:

When a Provider's value changes, every single component that called useContext for that context will re-render — even if the specific piece of data they use didn't change.

// Imagine this context holds three unrelated things
export function AppProvider({ children }) {
  const [user,    setUser]    = useState(null);
  const [cart,    setCart]    = useState({ count: 0 });
  const [notifs,  setNotifs]  = useState([]);

  return (
    <AppContext.Provider value={{ user, cart, notifs, setCart, setNotifs }}>
      {children}
    </AppContext.Provider>
  );
}

// Header only reads user.name
function Header() {
  const { user } = useContext(AppContext);
  return <h1>{user?.name}</h1>;
  // ⚠️ Will re-render when someone adds to cart, or gets a notification
  // Even though Header doesn't care about cart or notifs at all
}
perf-trap.jsx

The fix: split your contexts

One context per concern. Components subscribe only to the context they actually need.

// Split: three separate contexts
export const UserContext  = createContext(null);
export const CartContext  = createContext(null);
export const NotifsContext = createContext(null);

// Three providers, nested
export function AppProvider({ children }) {
  const [user,   setUser]   = useState(null);
  const [cart,   setCart]   = useState({ count: 0 });
  const [notifs, setNotifs] = useState([]);

  return (
    <UserContext.Provider value={{ user, setUser }}>
      <CartContext.Provider value={{ cart, setCart }}>
        <NotifsContext.Provider value={{ notifs, setNotifs }}>
          {children}
        </NotifsContext.Provider>
      </CartContext.Provider>
    </UserContext.Provider>
  );
}

// Now Header only re-renders when UserContext changes
function Header() {
  const { user } = useContext(UserContext); // ✓ isolated
  return <h1>{user?.name}</h1>;
}

// CartIcon only re-renders when CartContext changes
function CartIcon() {
  const { cart } = useContext(CartContext); // ✓ isolated
  return <span>{cart.count}</span>;
}

11React Context

When to use Context

✓ Context is great for

  • Auth / user info (changes rarely — on login/logout)
  • Theme (dark/light — almost never changes)
  • Language / locale setting
  • Feature flags or app config
  • Any global state that changes infrequently

✗ Context struggles with

  • Cart items (changes frequently)
  • Notification counts (real-time updates)
  • Form state across many fields
  • Any state that updates many times per second
  • State where different components need different "slices"

For high-frequency updates shared across many components — that's where Redux earns its place.

12Redux Toolkit

What Redux actually is

🏦
Redux is a bank

Your state is the bank's ledger. The only way to change the ledger is by submitting a transaction form (dispatching an action). The bank teller processes it using strict rules (the reducer). Any customer (component) can check their balance (useSelector) — but they use a window so they only see what they need, not the whole ledger. Every transaction is logged permanently (Redux DevTools).

Redux is a separate library — install it with npm. It has one global store for your whole app's state, and a strict, predictable pattern for how that state can change. Redux Toolkit (RTK) is the modern official way — it removes most of the old boilerplate.

The key difference from Context:

Context is a broadcast — change anything, all subscribers re-render. Redux has selectors — each component says "I only care about this specific value" and only re-renders when that specific value changes. This is what makes Redux better for high-frequency updates.

13Redux Toolkit

The 5 pieces you always write

Redux has more moving parts than Context. Here they all are — we'll go through each one deeply.

① createSlice()
Defines one domain of state + all its update logic
You tell Redux: here's the initial state, here are all the ways it can change. It gives you back actions and a reducer automatically.
Analogy: one department in the bank (savings dept, loans dept)
② configureStore()
Combines all slices into one global store
The bank itself. Combines all your department slices. Every component can read from it via useSelector.
Analogy: the bank building that contains all departments
③ <Provider store={store}>
Makes the store available to the whole app
Redux's own Provider component from react-redux. Wraps your app in main.jsx just like Context's Provider.
Analogy: the bank's address — how everyone knows where to go
④ useSelector()
Read a specific piece of state from the store
A hook you use in components. You pass it a function that picks exactly the piece you need. Only re-renders when that specific piece changes.
Analogy: the teller window — you ask for your balance, not the whole ledger
⑤ useDispatch()
Trigger a state change
Returns the dispatch function. You call it with an action to update the store. Like submitting a transaction form to the bank.
Analogy: the transaction form — fill it out, submit it, state changes
14Redux Toolkit

① createSlice — state and all its changes

A slice is one section of your global state, plus all the functions that can change it. You make one slice per domain: a user slice, a cart slice, an orders slice.

What does "slice" mean?
Your whole Redux state is one big object. A slice is literally one slice of that object — state.user is the user slice, state.cart is the cart slice. createSlice() is the RTK function that creates that slice for you.
// store/userSlice.js
import { createSlice } from '@reduxjs/toolkit';

const userSlice = createSlice({

  // "name" becomes the prefix for action type strings
  // e.g. 'user/setUser', 'user/clearUser'
  name: 'user',

  // What the state looks like at the start
  initialState: {
    data:    null,
    loading: false,
    error:   null,
  },

  // "reducers" — these are the ONLY ways state.user can change
  // Each function here becomes an "action creator" automatically
  reducers: {

    // setUser(state, action) — action.payload is what you pass when dispatching
    setUser: (state, action) => {
      state.data  = action.payload; // ← looks like mutation — it's SAFE in RTK
      state.error = null;           // RTK uses Immer to make this immutable under the hood
    },

    clearUser: (state) => {
      state.data = null;
    },

    setLoading: (state, action) => {
      state.loading = action.payload; // payload will be true or false
    },

    setError: (state, action) => {
      state.error = action.payload;
      state.loading = false;
    },
  },
});

// createSlice auto-generates action creators — these are what you dispatch
// userSlice.actions.setUser  → creates { type: 'user/setUser', payload: ... }
// userSlice.actions.clearUser → creates { type: 'user/clearUser' }
export const { setUser, clearUser, setLoading, setError } = userSlice.actions;

// Export the reducer — this is what configureStore needs
export default userSlice.reducer;

store/userSlice.js
About the "mutation" in reducers:

In vanilla Redux, you must return a new object — mutating state is forbidden. In RTK, you CAN write state.data = action.payload because RTK uses a library called Immer under the hood that converts your "mutation" into an immutable update automatically. This is one of RTK's biggest quality-of-life improvements.

Build the cart slice the same way

// store/cartSlice.js
import { createSlice } from '@reduxjs/toolkit';

const cartSlice = createSlice({
  name: 'cart',
  initialState: {
    items: [],
    count: 0,
    total: 0,
  },
  reducers: {
    addItem: (state, action) => {
      state.items.push(action.payload);   // again — looks like mutation, is safe
      state.count += 1;
      state.total += action.payload.price;
    },
    removeItem: (state, action) => {
      const index = state.items.findIndex(i => i.id === action.payload);
      if (index !== -1) {
        state.total -= state.items[index].price;
        state.items.splice(index, 1);
        state.count -= 1;
      }
    },
    clearCart: (state) => {
      state.items = [];
      state.count = 0;
      state.total = 0;
    },
  },
});

export const { addItem, removeItem, clearCart } = cartSlice.actions;
export default cartSlice.reducer;

15Redux Toolkit

② configureStore — combine everything

This is short. You take all your slices' reducers and combine them into one store. The key names you choose here become the keys on your state object.

// store/index.js
import { configureStore } from '@reduxjs/toolkit';
import userReducer  from './userSlice';
import cartReducer  from './cartSlice';
import ordersReducer from './ordersSlice';

export const store = configureStore({
  reducer: {
    user:   userReducer,    // state.user   → managed by userSlice
    cart:   cartReducer,    // state.cart   → managed by cartSlice
    orders: ordersReducer,  // state.orders → managed by ordersSlice
  },
});

// Your global state now looks like:
// {
//   user:   { data: null, loading: false, error: null },
//   cart:   { items: [], count: 0, total: 0 },
//   orders: { list: [], loading: false },
// }

17Redux Toolkit

④ useSelector — read from the store

This is how components read data from Redux. You pass it a selector function — a function that takes the entire state and returns the specific piece you need.

Why a function? Why not just useSelector('user.data')?
Because the function lets you do anything — compute derived values, combine multiple pieces, transform the data. It's powerful. And it's how React-Redux knows exactly what to compare when deciding whether to re-render.
import { useSelector } from 'react-redux';

// Selector function: takes ENTIRE state, returns ONE specific piece
// state is your whole { user: {...}, cart: {...}, orders: {...} } object

function Header() {
  // This component only cares about the user's name
  // It will ONLY re-render when state.user.data changes
  // Changing cart, orders, etc. will NOT re-render this
  const user = useSelector(state => state.user.data);

  return <h1>Hello, {user?.name}</h1>;
}

function CartIcon() {
  // Only cares about cart count — nothing else will trigger re-render
  const count = useSelector(state => state.cart.count);
  return <span className="badge">{count}</span>;
}

function CartTotal() {
  // You can compute derived values in the selector
  const total = useSelector(state =>
    state.cart.items.reduce((sum, item) => sum + item.price, 0)
  );
  return <span>Total: ${total.toFixed(2)}</span>;
}

using-useSelector.jsx
Common mistake — don't do this:

Never write useSelector(state => state) — that selects the entire store, so your component re-renders on every single dispatch, for any reason. Always select the smallest piece you need.

18Redux Toolkit

⑤ useDispatch — trigger a change

Reading is done by useSelector. Writing — changing state — is done by dispatching actions. useDispatch() gives you the dispatch function. You call it with one of the action creators your slice generated.

import { useDispatch } from 'react-redux';
import { addItem, removeItem } from './store/cartSlice';
import { clearUser } from './store/userSlice';

function ProductCard({ product }) {
  const dispatch = useDispatch();

  const handleAdd = () => {
    // dispatch takes an action object
    // addItem() creates: { type: 'cart/addItem', payload: product }
    dispatch(addItem(product));
    // ↑ Redux receives this, finds the matching reducer in cartSlice,
    // runs it, updates state.cart, and all cart subscribers re-render
  };

  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={handleAdd}>Add to cart</button>
    </div>
  );
}

function Header() {
  const user     = useSelector(state => state.user.data);
  const dispatch = useDispatch();

  const handleLogout = () => {
    dispatch(clearUser());   // clearUser() creates { type: 'user/clearUser' }
    dispatch(clearCart());   // can dispatch multiple actions
  };

  return <button onClick={handleLogout}>Logout</button>;
}

21Redux Toolkit

Async actions — API calls with Redux

So far, all our reducers are synchronous. But in the real world, you fetch data from an API. Redux Toolkit has createAsyncThunk for this.

Why can't I just do async stuff inside a reducer?
Reducers must be pure functions — no side effects. An API call is a side effect. createAsyncThunk gives you a place to put async logic that then automatically dispatches actions when it starts, succeeds, or fails.
// store/userSlice.js — adding async login
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// createAsyncThunk takes a name and an async function
// It automatically dispatches three actions:
// loginUser.pending  — when the fetch starts
// loginUser.fulfilled — when it succeeds (payload = what you return)
// loginUser.rejected  — when it fails (error)
export const loginUser = createAsyncThunk(
  'user/login',                          // action name prefix
  async ({ email, password }, thunkAPI) => {
    const response = await fetch('/api/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    });

    if (!response.ok) {
      return thunkAPI.rejectWithValue('Login failed'); // triggers .rejected
    }

    const data = await response.json();
    return data.user; // this becomes action.payload in .fulfilled
  }
);

const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, loading: false, error: null },
  reducers: {
    clearUser: state => { state.data = null; },
  },
  // extraReducers handles the thunk's auto-dispatched actions
  extraReducers: builder => {
    builder
      .addCase(loginUser.pending, state => {
        state.loading = true;
        state.error   = null;
      })
      .addCase(loginUser.fulfilled, (state, { payload }) => {
        state.data    = payload;   // payload = the user object we returned
        state.loading = false;
      })
      .addCase(loginUser.rejected, (state, { payload }) => {
        state.error   = payload;   // payload = 'Login failed'
        state.loading = false;
      });
  },
});

export const { clearUser } = userSlice.actions;
export default userSlice.reducer;

// ─────────────────────────────────────────
// In your component — dispatch it exactly like any other action
// ─────────────────────────────────────────
function LoginForm() {
  const dispatch = useDispatch();
  const loading  = useSelector(s => s.user.loading);
  const error    = useSelector(s => s.user.error);

  const handleSubmit = async (email, password) => {
    // dispatch returns a promise — you can await it
    const result = await dispatch(loginUser({ email, password }));

    // Check if it succeeded
    if (loginUser.fulfilled.match(result)) {
      // success — navigate away
    }
  };

  return <form onSubmit={handleSubmit}>...</form>;
}

22Decision

Context vs Redux — how to choose

Reach for Context when:

  • Data changes rarely (login/logout, theme toggle)
  • Small-to-medium app
  • No installation needed, simpler code
  • Only 2–3 components need the data
  • Example: theme, auth state, locale/language

Reach for Redux when:

  • Data changes frequently (cart, feed, notifications)
  • Medium-to-large app
  • You want Redux DevTools (time travel debugging)
  • Many unrelated components need the same data
  • You have complex async flows (API chains)
You can and should use both in one app:

Theme → Context (never changes). Auth user → Context (changes only on login/logout). Cart, orders, products → Redux (update constantly). This is the real-world pattern.
