Syntax & Structure
The grammar of Python — how the interpreter tokenises and parses your source code before execution even begins.
🧱 Why Indentation-as-Syntax?

Most languages use {} braces to delimit blocks. Python uses whitespace indentation as the actual syntax — not just a style choice. The Python tokeniser generates INDENT and DEDENT tokens when indentation changes, so the grammar sees them the same as { and } in other languages.

This was a deliberate design decision by Guido van Rossum: if indentation is mandatory for readability anyway, why not make it the syntax? The result is code where visual structure is logical structure — they can never diverge.

The rule: Python 3 requires consistent indentation within a block — all lines must use the same character (spaces or tabs, never mixed). PEP 8 mandates 4 spaces. The interpreter doesn't care how many spaces you use, only that you're consistent.

⚠️ TabError: Mixing tabs and spaces in Python 3 raises TabError immediately during tokenisation — before any code runs. In Python 2, tabs were treated as 8 spaces which caused silent bugs. Python 3 made it a hard error.
🔤 Comments, Docstrings & the Tokeniser

# comments are stripped by the tokeniser — they never reach the parser. Docstrings are different: they are actual string literals placed as the first statement of a module, function, class, or method, and they get stored in the object's __doc__ attribute at runtime.

This means help(func), IDE tooltips, and documentation generators like Sphinx all read docstrings from the live object. They are not comments — they are data.

Line continuation: Inside any bracket type ((), [], {}), line breaks are implicit. Outside brackets, use \ at line end. The implicit form is preferred because \ is fragile — a trailing space after \ breaks it silently.

Variables & Types
Python's type system is dynamic and strong. Variables are names — not boxes. Understanding this distinction is foundational.
🏷️ Names, Objects, and References — The Core Mental Model

In Python, variables are not containers. They are names (labels) that are bound to objects living in the heap. When you write x = 42, you are creating an integer object 42 in memory and binding the name x to it in the current namespace (a dict).

Assignment never copies data. y = x makes y point to the same object as x. You can verify with id(x) == id(y) — id() returns the memory address. This is called aliasing.

This matters enormously for mutable vs immutable objects. If two names point to the same mutable object (like a list), modifying through one name affects the other. Immutable objects (int, str, tuple, frozenset) cannot be changed, so aliasing is safe.

references.py
# Aliasing mutable objects — classic gotcha a = [1, 2, 3] b = a # b is NOT a copy — same list object b.append(4) print(a) # [1, 2, 3, 4] ← a is also changed! # To actually copy: b = a[:] # shallow copy via slice b = a.copy() # explicit shallow copy import copy b = copy.deepcopy(a) # deep copy — copies nested objects too # Immutable objects — aliasing is safe x = "hello" y = x y += " world" # creates a NEW string object, rebinds y print(x) # "hello" ← x is unchanged # Small int / string interning (CPython implementation detail) a = 256; b = 256 print(a is b) # True — CPython caches small ints -5..256 a = 257; b = 257 print(a is b) # False in most contexts (not guaranteed!) # Always use == for value equality, not 'is' # 'is' tests identity (same object), not equality
📦 Dynamic & Strong Typing

Dynamic means types are checked at runtime, not compile time. A variable can hold an int now and a string later — the name has no type, only the object does. This is why type(x) exists: it interrogates the object, not the variable.

Strong means Python does not silently coerce types. "1" + 1 raises TypeError. Unlike JavaScript, Python will not guess what you meant. This makes bugs explicit rather than silent.

Type hints (PEP 484) are annotations on variables and functions. At runtime they are completely ignored — Python doesn't enforce them. Their value is for static analysis tools (mypy, pyright, IDEs). Think of them as machine-readable documentation.

✅ Professional practice: Use type hints everywhere in production code. They make mypy --strict possible, which catches entire classes of bugs before runtime. The typing module provides Optional, Union, Literal, TypeVar, Protocol, TypedDict etc.

This is transparent to you — 2**1000 just works. The cost is that big integer arithmetic is slower than C int ops.

Operators
Operators in Python are syntactic sugar for method calls. Understanding this unlocks operator overloading in your own classes.
⚙️ Operators Are Dunder Method Calls

Every operator dispatches to a special method on the object. a + b calls a.__add__(b). If that returns NotImplemented, Python tries b.__radd__(a) (the reflected version). This is how 1 + MyClass() can work even though int doesn't know about your class.

Short-circuit evaluation: and and or do not return True/False. They return one of their operands. x or default returns x if truthy, else default. This is a common Python idiom for default values — but be careful: 0 or 5 returns 5, which may not be what you want when 0 is a valid value.

is vs ==: is checks identity (same memory address via id()). == checks value equality (calls __eq__). Never use is for comparing values — only for singletons like None, True, False.

operators_deep.py
# Arithmetic operators → dunder methods 5 + 3 # int.__add__(5, 3) → 8 5 / 2 # int.__truediv__(5, 2) → 2.5 (always float!) 5 // 2 # int.__floordiv__(5, 2) → 2 (floor toward -∞) -7 // 2 # → -4 NOT -3! (floors toward -inf, not toward 0) 5 % 2 # int.__mod__(5, 2) → 1 2 ** 32 # int.__pow__(2, 32) → 4294967296 # Short-circuit — returns an OPERAND, not bool print(0 or "fallback") # "fallback" (0 is falsy) print(5 or "fallback") # 5 (5 is truthy — stops) print(5 and "ok") # "ok" (both truthy — returns last) print(0 and "ok") # 0 (0 falsy — short-circuits) # Falsy values: 0, 0.0, 0j, "", b"", [], {}, set(), None, False # Everything else is truthy # Walrus operator := (assignment expression, Python 3.8+) # Assigns AND returns the value in one expression data = [1, 2, 3, 4, 5] if (n := len(data)) > 3: print(f"Large dataset: {n} items") # Without walrus, you'd need: n = len(data); if n > 3: ... # Common use: avoid calling expensive function twice while chunk := f.read(8192): # read() called once per iteration process(chunk)
⚠️ Floor division gotcha: -7 // 2 is -4, not -3. Python always floors toward negative infinity. C and Java truncate toward zero. If you need truncation (C-style), use int(-7 / 2) → -3. This affects modulo too: -7 % 2 is 1 in Python (not -1) because floor(-7/2) = -4 and -7 - (-4*2) = 1.
Fundamentals · 04
Strings
Strings are immutable sequences of Unicode code points. Python 3 strings are always Unicode — there is no "byte string" confusion from Python 2.
🌐 Unicode, Encoding, and What Strings Actually Are

A Python str is a sequence of Unicode code points (integers 0–1,114,111). It is not a sequence of bytes. The character 'é' is code point U+00E9. len('é') is 1 — one code point.

Encoding converts a string to bytes for storage/transmission. 'hello'.encode('utf-8') returns a bytes object. b'hello'.decode('utf-8') converts back. You must know the encoding to decode bytes into text — guessing is a bug.

CPython stores short strings using either Latin-1, UCS-2, or UCS-4 internally depending on the highest code point present (PEP 393 "flexible string representation"). This is an implementation detail — the API is always Unicode.

f-strings (Python 3.6+) are the fastest and most readable formatting method. They are compiled at parse time — the expression inside {} is evaluated and converted at runtime. They outperform % and .format() because there's no string parsing at call time.

strings_deep.py
# f-string format spec mini-language value = 1234567.891 print(f"{value:,.2f}") # 1,234,567.89 print(f"{value:e}") # 1.234568e+06 print(f"{0.375:.2%}") # 37.50% print(f"{42:08b}") # 00101010 (binary, zero-padded) print(f"{value!r}") # repr() — good for debugging print(f"{value = }") # value = 1234567.891 (debug =) # Encoding / decoding text = "café" raw = text.encode("utf-8") # b'caf\xc3\xa9' — 5 bytes, 4 chars back = raw.decode("utf-8") # "café" print(len(text), len(raw)) # 4 chars, 5 bytes # str.join is O(n) — NEVER concatenate in a loop with += # Bad: s = ""; for w in words: s += w (creates new string each time) words = ["hello", "world", "from", "python"] good = " ".join(words) # allocates once — use this # Interning: CPython may intern string literals that look like identifiers a = "hello"; b = "hello" print(a is b) # probably True (interned at compile time) a = " hi"; b = " hi" print(a is b) # probably False (space → not interned)

Control Flow
Python's control structures have subtleties that trip up developers coming from other languages — especially for-else, pattern matching, and the walrus operator.
🔀 for-else: The Most Misunderstood Feature

Python's for and while loops have an else clause that most developers don't know about. The else block runs when the loop completes normally — meaning it was not terminated by a break.

The canonical use case is searching: if you break out of the loop because you found something, the else does not run. If the loop exhausts without a break (nothing found), the else runs. This eliminates the common "found flag" pattern:

found = False; for x in items: if cond: found=True; break → replaced by a clean for...else.

match/case (Python 3.10+) is structural pattern matching — far more powerful than a switch statement. It matches on the structure of data: you can match tuples, dicts, objects by attribute, and use guards. It uses a different kind of equality — it's pattern matching, not just ==.

control_flow_deep.py
# for-else: else runs only if no break occurred def find_prime(numbers): for n in numbers: for i in range(2, int(n**0.5) + 1): if n % i == 0: break # composite — inner else skipped else: return n # inner loop completed → n is prime return None print(find_prime([4, 6, 7, 10])) # 7 # match/case — structural pattern matching (3.10+) def handle_command(command): match command.split(): case ["quit"]: return "Quitting" case ["go", direction] if direction in ("north","south","east","west"): return f"Moving {direction}" case ["pick", "up", item]: return f"Picked up {item}" case _: return "Unknown command" print(handle_command("go north")) # Moving north print(handle_command("pick up sword")) # Picked up sword # range() is a lazy sequence object, not a list r = range(0, 10_000_000) # instant — no allocation print(9_999_999 in r) # O(1) membership test via __contains__
🔬 range() internals: range is a class (not a function) that implements __len__, __getitem__, and __contains__. x in range(n) is O(1) — it computes membership arithmetically, never iterating. This is much faster than x in list(range(n)).

Functions
Functions are first-class objects in Python — they have a type (function), attributes (__name__, __doc__, __code__), and can be passed, stored, and returned like any other value.
🎯 Default Argument Gotcha — The Mutable Default Trap

This is one of Python's most notorious bugs-in-waiting. Default argument values are evaluated once at function definition time, not each time the function is called. If you use a mutable object (list, dict, set) as a default, all calls share the same object.

The canonical fix: use None as the default and create a new object inside the function body. This pattern is so common it's idiomatic Python.

Argument passing semantics: Python uses "pass by object reference" (sometimes called "pass by assignment"). The function receives a reference to the same object. You can mutate the object through that reference (and the caller sees it), but you cannot rebind the caller's variable (that only changes your local name).

functions_deep.py
# ── Mutable default bug ─────────────────────────────── def bad_append(item, lst=[]): # lst created ONCE at def time lst.append(item) return lst bad_append(1) # [1] bad_append(2) # [1, 2] ← bug! shares the same list bad_append(3) # [1, 2, 3] def good_append(item, lst=None): # correct pattern if lst is None: lst = [] lst.append(item) return lst # ── Argument flavors ────────────────────────────────── def strict( pos_only, # positional-only (before /) /, normal, # positional or keyword *, # everything after * is keyword-only kw_only, # must be passed as keyword kw_default = 10, ): return pos_only + normal + kw_only + kw_default # strict(1, 2, kw_only=3) → 16 # strict(pos_only=1, ...) → TypeError (pos_only is positional-only) # ── *args / **kwargs ────────────────────────────────── # *args collects extra positional args into a TUPLE # **kwargs collects extra keyword args into a DICT def show(*args, **kwargs): print(f"args = {args}") # tuple print(f"kwargs = {kwargs}") # dict show(1, 2, x=3, y=4) # Unpacking at call site (inverse of *args/**kwargs) def add(a, b, c): return a + b + c nums = [1, 2, 3] add(*nums) # same as add(1, 2, 3) add(**{"a":1,"b":2,"c":3}) # same as add(a=1, b=2, c=3)
✅ Design tip: Use / and * in your APIs. / makes positional-only params work correctly when internal param names might change. * forces keyword-only for params that are optional flags — callers must be explicit about what they're setting, making call sites self-documenting.

Scope & Closures
Scope in Python is determined statically at compile time based on where names are assigned — not at runtime. This is called lexical scoping.
🗺️ LEGB: How Python Resolves Every Name

When Python encounters a name like x, it searches four scopes in order:

  • Local (L) — the current function's local namespace (a dict stored in the frame)
  • Enclosing (E) — the local scope of any enclosing functions (for closures), searched outward
  • Global (G) — the module-level namespace (globals())
  • Built-in (B) — the builtins module (len, print, range, etc.)

If a name isn't found at any level, NameError is raised. The key insight: Python determines at compile time whether a variable in a function is local or not. If any assignment to a name exists in a function, the entire function treats that name as local. This is why code like the example below raises UnboundLocalError.

scope_deep.py
# UnboundLocalError — the most common scope trap x = 10 def broken(): print(x) # UnboundLocalError! Python sees x=x+1 below x = x + 1 # this assignment makes x LOCAL for the WHOLE function # so print(x) tries to read local x before assignment # Fix: declare intent def fixed_global(): global x # now x refers to module-level x x = x + 1 # ── Closures ───────────────────────────────────────── # A closure is a function that captures (closes over) free variables # from its enclosing scope. Those variables live in a 'cell' object. def make_adder(n): def adder(x): return x + n # n is a FREE variable — captured from make_adder return adder add5 = make_adder(5) add5(10) # 15 — make_adder's frame is GONE but n=5 lives in the cell # Inspect the closure print(add5.__closure__[0].cell_contents) # 5 print(add5.__code__.co_freevars) # ('n',) # ── Classic closure-in-loop bug ────────────────────── funcs = [lambda: i for i in range(5)] [f() for f in funcs] # [4, 4, 4, 4, 4] — all see i=4! # All lambdas close over the SAME variable i, not its value # Fix: capture value as default argument (evaluated at definition) funcs = [lambda i=i: i for i in range(5)] [f() for f in funcs] # [0, 1, 2, 3, 4] — correct # ── nonlocal ───────────────────────────────────────── def make_counter(start=0): n = start def tick(): nonlocal n # assigns to n in make_counter's frame, not local n += 1 return n return tick c1 = make_counter() c2 = make_counter(100) c1(); c1(); c1() # 1, 2, 3 c2() # 101 — independent state
⚠️ The loop closure bug is one of the most common Python interview questions and real-world bugs. Closures capture variables by reference, not by value. In a loop, there's only one loop variable — all closures share it. The fix (lambda i=i: i) uses a default argument, which is evaluated at definition time, capturing the current value.
Why does lambda: i for i in range(5) produce [4,4,4,4,4] when called?
A
lambdas don't support closures in loops
B
All lambdas close over the same i variable. By call time the loop has finished, so i=4
C
Python evaluates lambda bodies eagerly in list comprehensions
D
The loop variable is implicitly frozen at 4 for all lambdas

Data Structures
Python's built-in data structures are implemented in C and highly optimised. Knowing their time complexities is essential for writing performant code.
⏱️ Complexity You Must Know
  • list: append O(1) amortised, insert/delete at index O(n), index access O(1). Under the hood: dynamic array (like C++ vector).
  • dict: get/set/delete O(1) average, O(n) worst case (hash collisions). Python 3.7+ maintains insertion order. Implementation: open-addressing hash table with a separate compact array for ordering.
  • set: add/discard/membership O(1) average. Same hash table as dict but stores only keys. Use it for fast membership testing: x in big_set is O(1) vs x in big_list which is O(n).
  • deque: O(1) append/pop from both ends. Use instead of list when you need a queue — list.pop(0) is O(n), deque.popleft() is O(1).

Comprehension vs loop: List comprehensions are faster than equivalent for loops because they avoid the overhead of Python's bytecode dispatch for each iteration — the append is done in a tighter C loop internally.

OOP & Classes
In Python, classes are objects too — instances of type. Understanding this opens up metaclasses, descriptors, and the MRO.
🔗 The MRO — Method Resolution Order (C3 Linearisation)

When you call a method, Python searches the class hierarchy to find it. For single inheritance this is trivial, but Python supports multiple inheritance. The MRO is the ordered list of classes Python searches.

Python uses the C3 linearisation algorithm to compute the MRO. It guarantees: (1) subclasses appear before parents, (2) parents appear in the order listed in the class definition, (3) consistency across the hierarchy. Violations cause TypeError: Cannot create a consistent method resolution order.

Run MyClass.__mro__ or MyClass.mro() to inspect it. Always use super() instead of calling parent methods directly — it ensures proper MRO traversal in cooperative multiple inheritance.

🔧 Descriptors — The Mechanism Behind Properties

A descriptor is any object that defines __get__, __set__, or __delete__. When attribute access occurs on a class, Python's attribute lookup checks if the class or any parent has a descriptor for that name — and calls the descriptor's method instead of returning the object directly.

@property is just a built-in descriptor factory. classmethod and staticmethod are also descriptors. Understanding descriptors explains why obj.method returns a bound method — the function's __get__ is called, binding self.

Data descriptors (define both __get__ and __set__) take priority over the instance's __dict__. Non-data descriptors (define only __get__) are overridden by instance attributes.

oop_deep.py
# ── MRO example ────────────────────────────────────── class A: def greet(self): return "A" class B(A): def greet(self): return "B" + super().greet() class C(A): def greet(self): return "C" + super().greet() class D(B, C): # Diamond inheritance pass print(D.__mro__) # (D, B, C, A, object) print(D().greet()) # "BCA" — super() follows MRO, not just "parent" # ── Custom Descriptor ──────────────────────────────── class Validated: """Descriptor that validates value on set.""" def __set_name__(self, owner, name): self.name = name # called when class is created self.private = f"__{name}" def __get__(self, obj, objtype=None): if obj is None: return self # class-level access → return descriptor return getattr(obj, self.private, None) def __set__(self, obj, value): if not isinstance(value, str): raise TypeError(f"{self.name} must be str") setattr(obj, self.private, value.strip()) class User: name = Validated() # descriptor at class level email = Validated() def __init__(self, name, email): self.name = name # triggers Validated.__set__ self.email = email u = User(" Alice ", "alice@example.com") print(u.name) # "Alice" (stripped)
🔬 __slots__: By default, every instance has a __dict__ (a hash table) to store attributes — ~200 bytes overhead per instance. Adding __slots__ = ('x', 'y') replaces the dict with a fixed-size struct, saving 40–60% memory. Essential when creating millions of instances. Trade-off: you lose dynamic attribute assignment.

Error Handling
Python's exception system is part of normal control flow — not just for errors. Understanding exception chaining, context managers, and the else clause separates good Python from great Python.
🔗 Exception Chaining & the Traceback

Python 3 introduced implicit exception chaining: if an exception is raised while handling another exception, both are stored. raise X from Y (explicit chaining) sets X.__cause__ = Y. The traceback shows the full chain. raise X from None suppresses the context.

This matters when building library code: catch a low-level exception (e.g. OSError) and raise a domain-specific one. With proper chaining, users see both the high-level problem and the root cause.

The else clause runs only if the try block succeeded without exception — and crucially, any exception raised in else is not caught by the preceding except clauses. Use it to separate "the thing that might fail" from "actions to take on success".

exceptions_deep.py
# ── Exception chaining ─────────────────────────────── class DatabaseError(Exception): pass class RecordNotFound(DatabaseError): def __init__(self, table, pk): super().__init__(f"No {table} with pk={pk}") self.table = table; self.pk = pk def get_user(user_id): try: raw = db.query(f"SELECT * FROM users WHERE id={user_id}") except ConnectionError as e: raise DatabaseError("DB unavailable") from e # explicit chain if not raw: raise RecordNotFound("users", user_id) return raw # ── try/except/else/finally semantics ──────────────── def read_config(path): f = None try: f = open(path) # might raise FileNotFoundError except FileNotFoundError: return {} else: data = f.read() # only runs if open succeeded return parse(data) # exceptions here go UNCAUGHT (intentional) finally: if f: f.close() # ALWAYS runs, even on return # ── Context manager protocol ───────────────────────── # 'with' calls __enter__ and __exit__ # __exit__ receives exception info (type, value, traceback) # returning True from __exit__ SUPPRESSES the exception class Timer: import time def __enter__(self): self.start = self.time.perf_counter() return self # 'as' variable receives this def __exit__(self, exc_type, exc_val, tb): self.elapsed = self.time.perf_counter() - self.start print(f"Elapsed: {self.elapsed:.4f}s") return False # don't suppress exceptions with Timer() as t: sum(range(10_000_000))

Decorators & Generators
Decorators are a higher-order function pattern. Generators are suspended execution. Both are fundamental to Python's approach to metaprogramming and lazy evaluation.
🎨 Decorators: The Full Mental Model

@decorator is exactly equivalent to fn = decorator(fn). Nothing more. Any callable that takes a callable and returns a callable is a decorator. The @ syntax just applies it at definition time.

Why @wraps? Without it, your wrapper function replaces the original's __name__, __doc__, __module__, etc. This breaks documentation, debugging, and introspection. @wraps(func) copies those attributes to the wrapper. Always use it.

Decorator factories (decorators with arguments) require three levels of nesting: the factory function (takes config), the decorator (takes func), and the wrapper (takes func's args). The factory is called first, returns the decorator.

⚡ Generators: Coroutines Before Async

A generator function contains yield. When called, it returns a generator object immediately without executing any code. Each call to next() resumes execution until the next yield, then suspends again. The function's local state (variables, position) is preserved in the generator object's frame.

The protocol: Any object implementing __iter__ and __next__ is an iterator. A generator is a convenient way to create iterators without writing a class. StopIteration is raised when the generator function returns.

yield from does more than delegation. In a coroutine context, it also passes send() and throw() calls through. It's the basis of Python's original coroutine design before async/await.

Async / Await
asyncio implements cooperative multitasking on a single thread using an event loop. Understanding when it helps (and doesn't) is as important as knowing the syntax.
🔄 The Event Loop: What Actually Happens

When you await asyncio.sleep(1), your coroutine doesn't block — it suspends itself and tells the event loop "wake me up in 1 second." The event loop then runs other ready coroutines until yours is due to resume.

Crucially, only one coroutine runs at a time (no parallelism). The benefit is that I/O waits (network, disk) are handled by the OS without tying up a thread. For CPU-bound tasks, asyncio provides zero benefit — the GIL still limits you to one CPU, and you add scheduling overhead. Use ProcessPoolExecutor for CPU work.

asyncio.gather() submits all coroutines to the event loop and waits for all to complete. They run concurrently (interleaved), not in parallel. If one raises an exception, gather() cancels the others by default.

async context managers and async generators follow the same protocol as their sync counterparts — just with __aenter__/__aexit__ and __aiter__/__anext__. Use async with and async for.

async_deep.py
import asyncio # ── Concurrency vs Parallelism ──────────────────────── async def io_task(name, delay): print(f"→ {name} starting") await asyncio.sleep(delay) # simulates network/disk wait print(f"✓ {name} done") return name async def main(): # Sequential: 3 seconds total await io_task("A", 1.0) await io_task("B", 1.0) await io_task("C", 1.0) # Concurrent: ~1 second total results = await asyncio.gather( io_task("A", 1.0), io_task("B", 1.0), io_task("C", 1.0), ) # ["A", "B", "C"] # gather with return_exceptions=True — don't cancel others on error results = await asyncio.gather( io_task("X", 0.5), io_task("Y", 0.5), return_exceptions=True ) # Timeout on individual task try: result = await asyncio.wait_for(io_task("slow", 5.0), timeout=1.0) except asyncio.TimeoutError: print("Task timed out") # ── Running sync code in threadpool ────────────────── # Never call blocking code in a coroutine — it blocks the event loop import time async def safe_blocking(): loop = asyncio.get_event_loop() # run_in_executor offloads to ThreadPoolExecutor result = await loop.run_in_executor(None, time.sleep, 1) # Use ProcessPoolExecutor for CPU-bound work asyncio.run(main())
⚠️ The blocking call trap: Any synchronous blocking call inside a coroutine (e.g. requests.get(), time.sleep(), any heavy computation) blocks the entire event loop — all other coroutines freeze until it returns. Use async libraries: aiohttp instead of requests, aiofiles instead of open(), run_in_executor for unavoidable blocking calls.

Metaclasses & Dunder Methods
Metaclasses are classes for classes. type is the default metaclass — every class you define is an instance of type. This is the layer where frameworks like Django ORM, SQLAlchemy, and Pydantic operate.
🧬 How Classes Are Created: The Full Lifecycle

When Python executes a class statement, it: (1) executes the class body in a fresh namespace, (2) identifies the metaclass (default: type), (3) calls metaclass(name, bases, namespace) which returns the class object.

By customising the metaclass, you intercept class creation. This lets you: auto-register subclasses, enforce interface rules at class-definition time, modify class attributes before instances are created, and implement things like ORMs (where class User(Model) automatically creates a database table).

For most use cases, __init_subclass__ (Python 3.6+) and class decorators are simpler alternatives to full metaclasses and should be preferred.

# ── Metaclass basics ───────────────────────────────── # type(name, bases, dict) creates a class dynamically MyClass = type("MyClass", (object,), {"x": 42, "greet": lambda self: "hi"}) MyClass().greet() # "hi" — same as writing class MyClass: x=42; def greet: ... # ── Plugin registry via __init_subclass__ ──────────── class Serializer: """Base: any subclass with format= gets auto-registered.""" _registry: dict = {} def __init_subclass__(cls, format: str = None, **kw): super().__init_subclass__(**kw) if format: Serializer._registry[format] = cls @classmethod def get(cls, format): return cls._registry.get(format) class JsonSerializer(Serializer, format="json"): pass class YamlSerializer(Serializer, format="yaml"): pass Serializer.get("json") # JsonSerializer (auto-registered) # ── Full metaclass: enforce abstract methods ───────── class InterfaceMeta(type): def __new__(mcs, name, bases, namespace): cls = super().__new__(mcs, name, bases, namespace) # Find required methods from base classes required = set() for base in bases: required |= getattr(base, "_required", set()) missing = required - set(namespace) if missing and bases: # skip the base class itself raise TypeError(f"{name} must implement: {missing}") return cls
🔬 Pydantic, SQLAlchemy, Django ORM all use metaclasses or __init_subclass__ extensively. When you write class User(Base): name = Column(String), the metaclass intercepts this, reads the Column descriptors, and builds the SQL schema. When you write class Settings(BaseModel): name: str, Pydantic's metaclass reads the type hints and generates validators. The model looks like a class definition but actually registers complex runtime behaviour.

NumPy & Pandas
NumPy's speed comes from vectorised operations on contiguous C arrays. The Python loop is replaced by a tight C loop — typically 50–100× faster. Understanding this mental model is key.
🔢 Why NumPy is Fast: Vectorisation & Broadcasting

A Python list stores references to Python objects — each element is a heap-allocated object with type info, reference count, and value. A NumPy array stores raw numeric data in a contiguous C buffer — no Python overhead per element.

When you write a * 2 on a NumPy array, NumPy calls a compiled C function that iterates the buffer with SIMD instructions. No Python is executed per element. This is vectorisation.

Broadcasting is NumPy's rule for operating on arrays of different shapes. A scalar is broadcast across the entire array. A (3,1) array broadcasts against a (1,5) array to produce (3,5). The rule: dimensions are aligned from the right; if they differ, a size-1 dimension is stretched to match.

Pandas builds on NumPy. A DataFrame is essentially a dict of NumPy arrays (columns) sharing an index. Operations are vectorised. apply() with a Python function is slow — prefer built-in vectorised operations.

numpy_pandas_deep.py
import numpy as np import pandas as pd # ── NUMPY ═════════════════════════════════════════════ # Views vs copies — critical for performance and correctness a = np.arange(12).reshape(3, 4) b = a[:2, :2] # b is a VIEW — shares memory with a b[0, 0] = 999 # modifies a too! c = a[:2].copy() # explicit copy — independent # Broadcasting rules in action a = np.ones((3, 1)) # shape (3,1) b = np.ones((1, 4)) # shape (1,4) c = a + b # shape (3,4) — broadcast both dimensions # Boolean indexing — no Python loop data = np.random.randn(1000) outliers = data[np.abs(data) > 3] # automatic mask data[np.abs(data) > 3] = 0 # zero out outliers in-place # np.where: vectorised if/else result = np.where(data > 0, data, 0) # relu! # ── PANDAS ════════════════════════════════════════════ df = pd.read_csv("data.csv") # Chained assignment warning — use .loc for assignment df[df.age > 30]["score"] = 100 # SettingWithCopyWarning! df.loc[df.age > 30, "score"] = 100 # correct way # apply() is slow — prefer vectorised alternatives df["len"] = df["name"].apply(len) # slow — Python loop df["len"] = df["name"].str.len() # fast — vectorised # groupby + agg: powerful data summarisation summary = df.groupby("category").agg( mean_score=("score", "mean"), count=("score", "count"), max_age=("age", "max"), ).reset_index() # Method chaining — Pandas style result = (df .query("age > 25 and score > 80") .assign(grade=lambda x: pd.cut(x.score, bins=[0,60,80,100], labels=["C","B","A"])) .groupby("grade")["score"] .mean() )

ML Libraries

🧠 PyTorch: Tensors, Autograd & the Training Loop

PyTorch tensors are like NumPy arrays but with two additions: they can live on GPU and they support automatic differentiation (autograd). When you do math on tensors with requires_grad=True, PyTorch builds a computation graph. Calling .backward() traverses this graph in reverse to compute gradients.

The training loop is explicit in PyTorch (unlike Keras). This is intentional — you have full control over every step: forward pass, loss computation, gradient computation, gradient clipping, and parameter update. This makes debugging and research much easier than black-box APIs.

model.train() vs model.eval() controls BatchNorm and Dropout behaviour. In eval mode, BatchNorm uses running statistics (not batch), and Dropout is disabled. Always set the correct mode. Forgetting model.eval() at inference time causes inconsistent results.

pytorch_deep.py
import torch import torch.nn as nn # ── Autograd mechanics ─────────────────────────────── x = torch.tensor([2.0], requires_grad=True) y = x ** 3 + 2 * x # builds computation graph y.backward() # dy/dx = 3x² + 2 print(x.grad) # tensor([14.]) (3*4 + 2 = 14) # ── Custom nn.Module ───────────────────────────────── class TransformerBlock(nn.Module): def __init__(self, d_model, n_heads, ff_dim, dropout=0.1): super().__init__() self.attn = nn.MultiheadAttention(d_model, n_heads, dropout=dropout, batch_first=True) self.ff = nn.Sequential( nn.Linear(d_model, ff_dim), nn.GELU(), nn.Dropout(dropout), nn.Linear(ff_dim, d_model), ) self.norm1 = nn.LayerNorm(d_model) self.norm2 = nn.LayerNorm(d_model) self.drop = nn.Dropout(dropout) def forward(self, x, attn_mask=None, key_padding_mask=None): # Pre-norm formulation (better training stability) x_norm = self.norm1(x) attn_out, _ = self.attn(x_norm, x_norm, x_norm, attn_mask=attn_mask, key_padding_mask=key_padding_mask) x = x + self.drop(attn_out) # residual connection x = x + self.drop(self.ff(self.norm2(x))) return x # ── Production training loop ───────────────────────── def train_epoch(model, loader, optimiser, criterion, device, scaler): model.train() total_loss = 0 for batch in loader: xb, yb = batch[0].to(device), batch[1].to(device) with torch.autocast(device_type=device.type, dtype=torch.float16): preds = model(xb) # forward (mixed precision) loss = criterion(preds, yb) optimiser.zero_grad() # clear old gradients scaler.scale(loss).backward() # backward (scaled for fp16) scaler.unscale_(optimiser) nn.utils.clip_grad_norm_(model.parameters(), 1.0) # prevent explosion scaler.step(optimiser) scaler.update() total_loss += loss.item() return total_loss / len(loader)
✅ Best practice: Always use mixed precision training (torch.autocast + GradScaler). It's ~2× faster on modern GPUs with minimal code change. Always clip gradients — exploding gradients will silently corrupt training. Use AdamW (not Adam) — the weight decay implementation in Adam is mathematically incorrect.

🔑 SentenceTransformer: What it Actually Does

SentenceTransformer takes a transformer model (BERT, RoBERTa, etc.) and adds a pooling layer on top. The transformer produces one embedding vector per token. The pooling layer (mean pooling by default) averages these into a single fixed-size vector for the entire sentence.

The resulting dense vector is a point in a high-dimensional semantic space. Sentences with similar meaning are close together (high cosine similarity). This makes it ideal for: semantic search, clustering, paraphrase detection, deduplication, and RAG (Retrieval-Augmented Generation) pipelines.

Model selection matters: all-MiniLM-L6-v2 is fast (22M params, 384 dims) — good for prototyping. all-mpnet-base-v2 (109M params, 768 dims) is higher quality. BAAI/bge-large-en-v1.5 is SOTA for retrieval. Multilingual models (paraphrase-multilingual) handle 50+ languages.

Fine-tuning: Pre-trained embeddings may not capture your domain's vocabulary. Fine-tune using MultipleNegativesRankingLoss (for retrieval) or CosineSimilarityLoss (for similarity scores). Even a few hundred domain examples can substantially improve performance.

🤗 HuggingFace Transformers: The Full Picture

HuggingFace provides a unified API over hundreds of transformer architectures. The key abstraction is: AutoTokenizer + AutoModel. Given a model name string, they download the config and weights from the Hub and return the right class.

Tokenisation is often overlooked but critical. Tokenizers convert text to token IDs using vocabulary + subword algorithm (BPE, WordPiece, SentencePiece). Different models use different vocabularies — you must use the model's own tokenizer. The tokenizer also handles padding, truncation, special tokens ([CLS], [SEP]), and attention masks.

pipeline() is a high-level shortcut for common tasks. For production, use the lower-level AutoModel directly — it gives you control over batching, device placement, and output format.
